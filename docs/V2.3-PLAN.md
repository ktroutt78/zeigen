### D-02 — `MIC_SESSION_FAILED` salvage, gated on clean compositing

**The seam.** On the normal stop, `engine_stop` (`lib.rs:240-255`) gracefully stops the webcam segment (`webcam.stop_segment()`, writes `q\n`, finalizes the segment mp4) *before* the engine's `stopped` event triggers `recording_finalize` (`App.tsx:827-857`). On the error path, the engine self-resets to idle, so the frontend must **not** send `engine_stop` (it would produce a follow-on `INVALID_STATE`, the exact trap `recording_cleanup_local` was built to avoid). But `recording_finalize` and `recording_cleanup_local` both `.take()` the recording handle, so they can't be chained.

**Rust fix (idempotent finalize).** Make `recording_finalize` stop the webcam segment itself, before reading `segments()`:

- Currently `recording_finalize` moves `rec.webcam` into `webcam_opt` and reads `webcam.segments()`. Change it to bind the webcam mutably and call `webcam.stop_segment()` first, then read segments. `stop_segment` is already idempotent — on the normal path the segment was already stopped by `engine_stop`, so `self.current.take()` returns `None` and it's a no-op (`webcam.rs:69-79`). On the salvage path it finalizes the still-running last segment. This makes `recording_finalize` self-sufficient regardless of how it's reached.

**Frontend routing.** In the `error` case, branch on the code:

- `ev.code === "MIC_SESSION_FAILED"` → run the salvage flow, mirroring the `stopped` case (`App.tsx:827-857`): `setState("idle")`, `incReview()`, then `invoke<FinalizedRecording>("recording_finalize").then(info => openReview(\`review-${info.stamp}\`, info.scratch_mp4_path, decReview)).catch(err => { setError(...); decReview(); })`. Do **not** call `engine_stop`; do **not** call `recording_cleanup_local`. The engine has already finalized the partial screen file (D-04 semantics: "finalizes whatever has been written so far, then emits `error`"), so the file exists by the time the frontend sees the event.
- All other codes → unchanged discard path: `recording_cleanup_local` + `setState("idle")`.

The salvage branch reuses `recording_finalize`, `openReview`, and the review window unchanged — no save/edit change, no review-window change. The only new logic is the error-path branch and the idempotent `stop_segment` in finalize.

**The hard gate (DONE-WHEN, exercised not asserted).** Trigger `MIC_SESSION_FAILED` through the full Tauri app with webcam active and confirm the partial composites cleanly and opens in review:

- Launch the app with `ZEIGEN_FORCE_MIC_SESSION_ERROR=1` in the environment (it propagates to the spawned engine — `engine.rs` `Command::new` inherits parent env). The engine fires `MIC_SESSION_FAILED` ~100ms after `started` (per V2.2-REPORT), so the partial is short (~100ms of screen + ~100ms of webcam segment).
- Start a recording **with a webcam selected** (exercises `composite::composite`, the real risk).
- Expected: banner shows the friendly `MIC_SESSION_FAILED` message; the partial composites without error; the review window opens with a playable file.
- **Pass → salvage ships.** **Fail (composite chokes on the short/odd partial) → fall back to discard for `MIC_SESSION_FAILED`** (route it to `recording_cleanup_local` like other codes, change the D-01 string to the "couldn't be recovered" wording), and record the fallback in the report. No half-salvage that opens a broken review.
- Also smoke the webcam-inactive case (no segments → `recording_finalize` uses the screen mp4 directly): confirm it opens too. The webcam-active composite is the load-bearing assertion.

### D-04 — orphan webcam-ffmpeg parent-death protection (the load-bearing fix)

The webcam ffmpeg is a child of the **Tauri/Rust process** (`webcam.rs:37-62`), not the Swift engine. The existing `Drop` (`webcam.rs:90-97`) and `stop_segment` (`:69-79`) cover graceful teardown, but neither runs when the Tauri process dies ungracefully (crash / force-quit / SIGKILL) — the 187GB scenario. macOS has no `PR_SET_PDEATHSIG`, and a Rust-side supervisor thread dies with the process under SIGKILL, so the death-signal **must live in a separate process**.

**Mechanism — detached sibling watchdog (parent-PID poll, CONTEXT D-04 primary).** Keep ffmpeg a direct Rust child (preserves the `q\n`-to-stdin graceful stop and the `Drop`-kill fast path). Additionally spawn a tiny `/bin/sh` watchdog that watches **both** the ffmpeg PID and the Tauri PID and kills ffmpeg if the Tauri PID dies first:

```rust
let tauri_pid = std::process::id();
let ff_pid = child.id();
let watchdog = Command::new("/bin/sh")
    .arg("-c")
    .arg(format!(
        "while kill -0 {ff} 2>/dev/null && kill -0 {tauri} 2>/dev/null; do sleep 1; done; \
         kill -9 {ff} 2>/dev/null",
        ff = ff_pid, tauri = tauri_pid,
    ))
    .stdin(Stdio::null()).stdout(Stdio::null()).stderr(Stdio::null())
    .spawn()
    .ok(); // best-effort; absence must not block recording
```

- **Crash/SIGKILL of Tauri:** only the Tauri PID receives the signal; its children (ffmpeg, watchdog) reparent to launchd and survive. The next poll sees `kill -0 {tauri}` fail, the loop exits, `kill -9 {ff}` stops the orphan. Disk reclaims once ffmpeg dies.
- **Graceful stop:** `stop_segment` writes `q\n`, waits ffmpeg; ffmpeg exits; the watchdog's `kill -0 {ff}` fails and it self-exits within ≤1s. To avoid blocking the stop path and to reap immediately, `stop_segment` (and `Drop`) also `kill()` + `wait()` the stored watchdog handle directly.
- Store the watchdog in the struct: `watchdog: Option<Child>`. Reap in both `stop_segment` and `Drop` so no zombie accumulates on the normal path; on Tauri crash, launchd reaps the reparented watchdog.

Poll interval 1s (consistent with the D-14/D-15 1.0s timeouts). PID-reuse caveat: a vanishingly small window where the Tauri PID could be reused before the watchdog notices — negligible on macOS (slow PID reuse); note it inline. kqueue `EVFILT_PROC` (CONTEXT D-04 fallback) only if the poll proves flaky.

**Both PIDs are captured at spawn time, not resolved later.** `format!` bakes the literal integer values of `std::process::id()` (Tauri) and `child.id()` (ffmpeg) into the `-c` string, so the shell watches fixed numbers decided at spawn. This is load-bearing: a watchdog that tried to discover the parent PID after launch (e.g. a `getppid`-style lookup) would, by the time the parent is dead, watch nothing — the value must be frozen while the parent is still alive.

**Test helper — `harness/orphan-kill-test.sh`.** The SIGKILL case can't be fully automated (starting a webcam recording needs the React UI), so the human starts the recording and the script does the deterministic kill+assert half. Shape:

```text
harness/orphan-kill-test.sh [--graceful]
```
1. Resolve the Tauri app PID (`pgrep -f` the app binary, or take as arg) and the webcam ffmpeg PID (`pgrep -f avfoundation`).
2. Assert the webcam ffmpeg is running; if absent, abort with "no webcam ffmpeg found — is a webcam recording active?" (the script cannot verify a *physical* camera is producing frames — the human confirms the webcam was actively capturing at kill time).
3. Record start time. Default: `kill -9 <tauri_pid>` (ungraceful — `Drop`/`stop_segment` do NOT run). `--graceful`: send a normal quit instead (contrast control — proves the test is catching the *ungraceful* path specifically, since graceful is already handled by `Drop`).
4. Poll `kill -0 <ff_pid>` every 0.2s up to 3s; on disappearance record elapsed → PASS; if still alive at 3s → FAIL (orphan survived).
5. Print a one-line result for the c2 commit body: e.g. `SIGKILL: ffmpeg <pid> gone <N>ms after Tauri kill -9 — PASS`.

### G1 — synthetic salvage test broken since V2.2 c3 (engine-side, deferred)

The `ZEIGEN_FORCE_MIC_SESSION_ERROR=1` fault injection (added V2.2 c3 `8234321`, `RecordingSession.swift:283-295`) was c1's automatable proxy for the salvage path. G1 verification proves it is structurally broken on slow-init devices.

The synthetic schedules `DispatchQueue.global().asyncAfter(deadline: .now() + 0.100)` INSIDE `RecordingSession.start()`. The closure calls `onFatalError` → `Task { await self.handleFatalError(err) }` onto the Engine actor. `handleFatalError` guards `state != .idle, let s = session`. But in `Engine.handleStart` (`Engine.swift:185-204`), `session = newSession` and `state = .recording` are set AFTER `try await newSession.start()` returns. When SCK initialization exceeds 100ms (common on iPhone Continuity ~200-400ms; observed reliably on built-in only when first samples land in <100ms), the synthetic fires while still inside `start()`, the guard fails on `.idle`/`nil`, and `emit(.error)` is silently skipped. The fault has been a no-op since the V2.2 c3 landing commit.

Diagnostic: inline probe (`V2.3-G1-PROBE force-env=... captureSessionNonNil=... synthetic ...`) in RecordingSession.swift, exercised against iPhone+iPhone, observed `synthetic SKIPPED (already fired or cutoff set)`. Probe reverted before commit.

### c3.S1 (`aedb5ea`) — missing-screen.mp4 dead-review fix

**Bug.** Phase 15 c3 deferred composite to export and replaced composited review with a dual-stream player reading `sources/screen.mp4` directly. AVAssetWriter (`RecordingSession.swift:684-687`) only opens its output on `writer.startWriting()`, gated on both first-audio AND first-video samples arriving. When `handleSessionRuntimeError` (`:789`) fires `MIC_SESSION_FAILED` in the pre-writer-start window (iPhone Continuity device conflict, USB mic disconnect during init), `started == false` → `tearDownAfterFatalError` skips `finishWriting` → `screen.mp4` is never created. c1's salvage path unconditionally set `screen_path = sources_dir.join("screen.mp4")` (`lib.rs:412`) and returned `Ok`. App.tsx opened review against a non-existent file: black canvas, `onLoadedMetadata` never fires, NR pipeline errors, empty timeline. The friendly MIC_SESSION_FAILED banner ("...saved up to that point.") lied.

**Fix.** `recording_finalize` checks the resolved screen path before construction, bails with `Err("RECORDING_FAILED_BEFORE_START")` if absent. Both branches (`sources/screen.mp4` for webcam case, `scratch_mp4_path` for no-webcam-with-mic). Check fires BEFORE concat so the bail path doesn't leave an orphan `webcam.mp4` in scratch. Webcam ffmpeg child reaped via `WebcamSegmenter::Drop` on Err return; scratch dir collected by the 24h launch sweeper.

App.tsx salvage catch refines the friendly MIC_SESSION_FAILED banner copy on sentinel-bail: "Recording couldn't start. Please try again." Cause-neutral — the trigger may be device conflict, hot-swap, or driver glitch, not specifically the mic. Engine's `MIC_SESSION_FAILED` message field is a localized AVFoundation `NSError` description (i18n-variable, not stable across macOS versions) so message-pattern-matching for cause-specific copy is fragile.

### c3.S2 (`85ef7bb`) — stopped+error actor-reentry race → data loss

**Bug** (surfaced while diagnosing G1's apparent no-fire behavior). Swift actors are reentrant by default: an `await` inside an actor method releases the actor, allowing pending Tasks queued on it to run. When `RecordingSession`'s `handleSessionRuntimeError` fires during `Engine.handleStop`'s `await s.stop()` suspension, the pending `onFatalError → handleFatalError` Task runs INSIDE `handleStop`'s await. Both paths complete: handleFatalError emits `.error(MIC_SESSION_FAILED)`, sets `state = .idle, session = nil`; handleStop's `s.stop()` returns (RecordingSession's stop idempotency on already-torn-down state), session-stats logs, `emit(.stopped)` fires. **Both events reach App.tsx for the same recording.**

App.tsx (pre-c3.S2) handled this asymmetrically: stopped → `recording_finalize`; error → `recording_finalize` for MIC_SESSION_FAILED only, else `recording_cleanup_local`. Both Tauri commands race for the Rust state `Mutex<Option<ActiveRecording>>` via `.take()`. When a non-MIC error chained the cleanup branch (e.g. follow-on `INVALID_STATE` from `handleStop`'s guard after the actor reached `.idle`), `cleanup_local` won the race, took state, killed webcam (no-op since `engine_stop` already did), returned `Ok` — never ran concat. Recording silently discarded. The stopped path's subsequent `recording_finalize` returned `Err("no active recording")` → pre-c3.S2 catch surfaced that string as the banner.

Confirmed in field 2026-06-08 with iPhone+iPhone Continuity: 4s `webcam-00.mp4`, ~66ms `screen.mp4` (writer-start fired before the Continuity drop), no `webcam.mp4` (concat never ran), "no active recording" banner. Real data loss on what to the user was a normal Stop click.

**Fix.** Route ALL engine error codes through `recording_finalize` (eliminating the cleanup_local-vs-finalize split). Both event handlers race on the same `.take()` and both try to SAVE. Whichever wins runs concat and opens review; the loser bails with `"no active recording"` via the `.take()` + `?`-propagation at `lib.rs:380-388`. Race-loser safety verified: MutexGuard is held across `.take()`, `?`-propagation on None happens BEFORE any field extraction, file touch, or subprocess spawn — **zero side effects on bail.**

c3.S1's sentinel makes `recording_finalize` safe to call universally. The catch refines banner copy only for `MIC_SESSION_FAILED` (the only friendly map entry that lies on sentinel-bail); other codes' friendly map text already describes the failure accurately. The `"no active recording"` race outcome is now a silent no-op + `decReview()` in both catches, leaving the originating user-facing banner intact (e.g. the watchdog's "Recording engine didn't respond..." stays clean rather than being clobbered by the technical race string).

`recording_cleanup_local` becomes unused on the frontend. Tauri command + registration left intact for a separate dead-code-removal commit after field validation.

**How verified.**
