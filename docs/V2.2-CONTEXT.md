## Implementation decisions

### D-01: Clock verification on every `start`, per-recording AVCaptureSession (GA1)

AVCaptureSession is constructed fresh per `start` command and destroyed after `stop` — same lifecycle as the spike's `TrackBRunner`. No process-singleton, no input swapping between recordings. Permissions prompted once at engine startup (D-08); session lifecycle is otherwise per-recording.

On every `start`, after both `stream.startCapture()` and `session.startRunning()` complete, verify time-base parity:

- Snapshot `host_now_1 = CMClockGetTime(CMClockGetHostTimeClock())`.
- Snapshot `session_uptime = CMClockConvertHostTimeToSystemUptime(host_now_1)`.
- Snapshot `host_now_2 = CMClockGetTime(CMClockGetHostTimeClock())`.
- Confirm `session.synchronizationClock` is non-nil and that conversion round-trips within a small tolerance.

If verification fails, emit `error` with code `CLOCK_MISMATCH` and abort the recording (writer never starts; SCStream/session torn down). Verification success is silent — no IPC event, no stderr noise beyond the engine's normal start logs.

The verification function is exercised in test by `ZEIGEN_FORCE_CLOCK_MISMATCH=1` (D-12), the engineering-only fault-injection switch. The contract surface is just the error code.

Reference parity check (`session.synchronizationClock === CMClockGetHostTimeClock()`) is explicitly NOT the test — V2.1 proved the two are different CMClock object references that nonetheless tick in the same time base.

### D-02: Writer-start alignment (GA2)

Two-pipeline writer-start with max-origin:

- `firstVideoPTS` and `firstAudioPTS` start `.invalid`.
- On each sample arrival, record the pipeline's first PTS if not yet set.
- Gate `writer.startWriting()` + `writer.startSession(atSourceTime:)` until both `firstVideoPTS` and `firstAudioPTS` are valid.
- Session start origin: `max(firstVideoPTS, firstAudioPTS)`. The earlier pipeline's pre-origin samples are dropped (not buffered, not trimmed). Set `sessionStartPTS = max_origin` so `elapsed_s` computation continues from a sensible base.

The MP4's audio prefix collapses to ~0; the V2.1 startup_gap_ms (80-243ms) becomes irrelevant to the saved file.

**`started` IPC event timing.** Fires at `stream.startCapture()` + `session.startRunning()` return, BEFORE writer-start. Matches v1.0's behavior — the user sees the red-dot indicator without waiting for the (~tens of ms) writer-start window. The MP4 origin is internal-only.

**First-audio-sample timeout.** If `firstAudioPTS` is still `.invalid` 250ms after `firstVideoPTS` first becomes valid, fail the recording with `error` code `MIC_NO_FIRST_SAMPLE`. Tear down SCStream + AVCaptureSession + writer; emit the error; engine stays running (returns to idle). 250ms is tighter than my initial 500ms suggestion — a mic that hasn't produced a single sample 1/4 second after video started is broken, not slow.

Rationale: silently capturing video-only when mic is dead produces exactly the kind of "ships broken file" failure v2 sets out to eliminate. Failing fast is honest. The user can `start` again if they un-break their mic.

**Pause math under max-origin.** `totalPausedDuration` continues to be subtracted from `rawPTS` in `append()` per v1.0. The shift only moves `sessionStartPTS` from `first_sample_pts` to `max(first_video_pts, first_audio_pts)` — both video and audio samples are remapped against the same `totalPausedDuration` offset because they share host time. The output timeline stays gapless. PLAN includes an explicit walkthrough.

### D-03: Symmetric teardown via PTS cutoff clamp (GA3)

At `stop()` entry, before initiating teardown:

- Snapshot `cutoffPTS = CMClockGetTime(CMClockGetHostTimeClock())`.
- Mark a `stopping` flag (or store cutoffPTS as the gate; both shapes are equivalent).

In `append()`, reject any sample whose PTS > cutoffPTS — both video and audio handlers honor the same cutoff. This eliminates the asymmetry deterministically regardless of which handler queue the late sample arrives on (the AVCaptureAudioDataOutput delegate may still fire a few times after `session.stopRunning()` due to in-flight queue draining; SCStream's `stopCapture` await window similarly produces late video).

Then teardown:
- `await stream.stopCapture()` (awaits in-flight SCK callbacks).
- `session.stopRunning()` (sync; queue may drain after).
- `videoInput.markAsFinished()` + `audioInput.markAsFinished()`.
- `await writer.finishWriting`.

The order between `stopCapture` and `stopRunning` becomes less load-bearing because the cutoff clamp gates the writer regardless. Spike's order (SCK first, then session) is fine; reverse order is also fine.

### D-04: IPC contract — three new error codes, no new events, no new fields (GA4)

Three new error codes added to `docs/IPC-SPEC.md`:

- **`CLOCK_MISMATCH`** — D-01 verification failure on `start`. Recording does not begin; engine stays running.
- **`MIC_NO_FIRST_SAMPLE`** — D-02 250ms timeout. Recording teardown; engine stays running.
- **`MIC_SESSION_FAILED`** — D-07-adjacent. AVCaptureSession's `AVCaptureSessionRuntimeError` notification during recording representing a genuine session failure (not a hot-swap; see D-07 for the distinction). Engine finalizes whatever has been written so far, emits `error`, returns to idle.

Forcing function (per V2-CHARTER §"In-scope" IPC clause and V2-RISKS.md R3 fallback): these are new failure modes that did not exist in v1.0's architecture (clock verification did not happen, AVCaptureSession did not exist mid-recording, mic-no-first-sample could not occur with SCK mic). Mapping all three to existing `WRITER_FAILED` or `INTERNAL` codes loses diagnostic information V2.4's parity-gate verification depends on. The error-code surface is the minimum widening that preserves the cascade contract.

No new events. No new fields on `stopped`. No structural changes to existing payloads. `IPC-SPEC.md` updates land in the commit that introduces each code (per V2-RISKS.md R3 "docs/IPC-SPEC.md updates in the same commit").

### D-05: Two sample handlers, shared `append(_, isVideo:)` (GA5)

Mirror the spike literally:

- `stream(_:didOutputSampleBuffer:of:)` handles `.screen` from SCStream → `append(buffer, to: videoInput, isVideo: true)`.
- `captureOutput(_:didOutput:from:)` (AVCaptureAudioDataOutputSampleBufferDelegate) handles audio from AVCaptureSession → `append(buffer, to: audioInput, isVideo: false)`.

Shared `append()` retains its lock-based critical section: pause check → writer-start gate (D-02) → PTS cutoff check (D-03) → pause-offset adjustment → `input.append`. Per-pipeline first-PTS bookkeeping happens inside the critical section.

No actor refactor. The lock is the same lock the v1.0 code already uses; widening it to cover two-pipeline writer-start gating is a small additional cost, not a fundamental restructure. If the lock-juggling turns out painful during c2 implementation we revisit, but starting with the spike's shape is the least-surprise choice.

### D-06: Pause invalid before writer-start (GA6)

The IPC state machine gains an implicit sub-state: "recording, but writer-start not yet fired" (the ~tens-to-hundreds-of-ms window between `startCapture`/`startRunning` and gated `startSession`). Pause command during this window returns `error` with code `INVALID_STATE`.

The window is short enough that a user pressing pause within it is a corner case. Erroring is more honest than buffering the pause and applying it later (which adds state machine complexity for no real-world benefit).

Resume is moot in this discussion — resume from a "writer-not-started" state can't happen because pause isn't accepted.

### D-07: Mic hot-swap — continue video-only, `MIC_DROPPED` to stderr (GA7)

Distinguish two failure modes for AVCaptureSession during recording by notification type:

1. **`AVCaptureSessionWasInterrupted`** → mark `audioInput.markAsFinished()` early, let SCStream + writer continue with video only. Log `MIC_DROPPED reason=<reason>` to stderr. No IPC event.
2. **`AVCaptureSessionRuntimeError`** → emit `MIC_SESSION_FAILED` IPC error, finalize MP4, return to idle. D-04's third new code.

The bucket boundary is the notification type itself, not a curated list of cause codes within each notification. AVError's taxonomy is broad enough that enumerating illustrative causes here would read as a contract c3 implementation must honor — better to land on the two-bucket rule and let c3 UAT surface any case that falls into the wrong bucket. A misbucketed case is a CONTEXT amendment, not a c3 implementation surprise.

The resulting MP4 in case (1) has a shorter audio track than video track on purpose. This is NOT the V2 V−A drift we set out to eliminate — that was clock divergence on a successful recording. This is "mic genuinely died mid-record, we kept what we could." The two are distinguishable in practice: clock-divergence drift is sub-second and present on every recording; hot-swap drift is multi-second-to-minutes and only on recordings where mic actually disconnected.

IPC event for the hot-swap case is **deferred to V2.3** (when Tauri wires Rust-side handlers and the UX question "should the UI tell the user mic dropped mid-record" gets answered). For V2.2, stderr-only keeps the IPC contract minimally widened.

### D-08: Permission pre-request at engine startup (GA8)

Before the `ready` event, `main.swift` pre-requests:

- `CGRequestScreenCaptureAccess()` — blocking. Denial → emit `error` with code `PERMISSION_DENIED`, exit non-zero.
- `AVCaptureDevice.requestAccess(for: .audio)` — async, wrapped in continuation. Denial → emit `error` with code `PERMISSION_DENIED`, exit non-zero.

Both must succeed before `ready` fires. The Rust side already handles a slow `ready` (engine takes ~hundreds of ms to start anyway); the additional prompt-or-cached-check latency is acceptable.

Adopts the spike's pattern (`spikes/v2-unified-capture/Sources/spike/main.swift:166-175`) verbatim. The v1.0 engine relies on SCStream + SCK mic to prompt lazily on first `startCapture`; V2.2 moves both prompts to startup so AVCaptureSession's failure modes never have to surface a missing-mic-permission case mid-recording.

Camera permission is NOT pre-requested. V2.2 does not touch webcam; camera permission stays under ffmpeg's lazy-prompt-on-start-of-camera-process flow.

### D-09: Phase 12 limiter preservation — move call site only

V2.2 preserves `applyLimiterInPlace` (`RecordingSession.swift:382-417`) exactly as-is. The Phase 12 c2 soft-knee limiter (`-1 dBFS`, tanh knee, mono float PCM) is a settled behavior; V2.2 changes WHERE it's called, not what it does.

- v1.0 site: `stream(_:didOutputSampleBuffer:of:)` `.microphone` branch.
- V2.2 site: `captureOutput(_:didOutput:from:)` (AVCaptureAudioDataOutputSampleBufferDelegate).

The AVCaptureAudioDataOutput audio settings (D-05 spike inheritance) force mono float PCM 48kHz — matching the limiter's expected input shape (`UnsafeMutablePointer<Float>`). No conversion code needed.

Flagged explicitly here so c2 implementation does not "while we're here" refactor limiter behavior. Any limiter change is a future-phase decision per the V2-CHARTER out-of-scope clause.

### D-10: Standalone drift harness at `harness/v2-drift/`

Top-level `harness/` directory, peer to `spikes/`. The harness:

- Drives the engine binary via line-delimited JSON over stdin/stdout (the existing `docs/IPC-SPEC.md` contract; no Track B in IPC).
- Per take: spawn engine → wait for `ready` → send `enumerate` → send `start` → wait 30s → send `stop` → consume `stopped` → kill engine.
- Per take post-processing: run Phase 13's ffprobe invocation, compute `abs_drift_ms`, `audio_start_ms`, wall-clock duration.
- 10 takes total, varied across at least 2 displays (primary + external) per V2.4 §"Verification matrix" pattern.
- Output: `harness/v2-drift/output/results.csv` (per-take measurements) + `output/meta.csv` (host metadata: macOS version, SDK, engine commit hash).
- `harness/v2-drift/output/` added to `.gitignore` in c1.

Harness outlives V2.2 — V2.4's drift verification reuses it. Putting it under `spikes/v2-unified-capture/` would violate Charter §"Branch hygiene" ("spikes can be deleted post-merge; they are not load-bearing"). Top-level is honest about the lifetime.

Implementation is a shell driver + Python or shell post-processor — light scaffolding, not a Swift CLI. Mirror `spikes/v2-unified-capture/run.sh`'s shape; PLAN locks the exact form.

### D-11: c2 acceptance bar = `abs_drift_ms < 100ms` on 10/10 takes (NOT < 17ms)

The V2-CHARTER DONE bar (`abs_drift_ms < 17ms`) is V2.4's gate. V2.2's acceptance bar is one order of magnitude better than v1.0 and inside the `end_time_drift_ms` envelope V2.1 measured:

- v1.0 baseline (Phase 12/13): `abs_drift_ms` up to **644ms**.
- V2.1 spike `end_time_drift_ms`: median 62-70ms, max 87ms on primary, 71ms on external.
- V2.2 c2 bar: **`abs_drift_ms < 100ms` on 10/10 takes**. Order-of-magnitude improvement vs v1.0; within the `end_time_drift_ms` envelope V2.1 predicted; AAC priming residual (R2) explicitly outside this bar.

Per SPIKE-REPORT.md §"V2.2 carryover" item 2 ("Symmetric teardown"): "With writer-start alignment + symmetric teardown, the residual is dominated by AAC frame quantization (AAC frames at 48kHz are ~21ms; last-frame tail is 0-21ms). That residual is R2 territory and V2.4 measures it." V2.2 c2 landing under 100ms confirms the V2.1 prediction held in production; landing under 17ms is a pleasant surprise but not the contract.

If c2 lands at ~60ms it is exactly the spike's prediction and V2.2 is done correctly. If c2 lands at ~30ms, R2's residual may already be smaller than V2.4 assumed — note in V2.2-REPORT.md, defer the gate-close decision to V2.4. If c2 lands above 100ms on any take, c2 doesn't commit — investigate.

The decoupling between V2.2's bar and V2.4's bar is explicit: forcing 17ms at c2 conflates "engine ships the architectural fix" with "engine ships the encoder fix." Track B is the former; R2 is the latter.

### D-12: Engineering-only fault-injection env vars for unreachable error codes

Two new error codes (D-04) cannot be reliably triggered through real hardware or real misconfiguration:

- `CLOCK_MISMATCH` — would require an SDK regression in `CMClockConvertHostTimeToSystemUptime` or a synthetic broken-clock environment.
- `MIC_SESSION_FAILED` — would require an `AVCaptureSessionRuntimeError` notification that is not also an `AVCaptureSessionWasInterrupted` (per D-07's bucket boundary). No reliable way to provoke this on real hardware short of breaking the mic device driver.

Both get parallel engineering-only fault-injection environment variables:

- **`ZEIGEN_FORCE_CLOCK_MISMATCH=1`** — D-01's verification function skips the host-time round-trip and unconditionally fails. Exercises the `CLOCK_MISMATCH` error path on `start`.
- **`ZEIGEN_FORCE_MIC_SESSION_ERROR=1`** — D-07's runtime-error handler is invoked synthetically once AVCaptureSession reaches `isRunning == true`, with a constructed `AVError` payload. Exercises the `MIC_SESSION_FAILED` error path mid-recording (via the `onFatalError` callback, same path a real runtime error would take).

Symmetric coverage: three new error codes (D-04), three exercised paths. The third — `MIC_NO_FIRST_SAMPLE` — is triggered through a code-level smoke (temporarily disabling the audio delegate; see PLAN c2 done-when). Three codes, three exercised paths. The alternative "code review only" for `MIC_SESSION_FAILED` is rejected: runtime coverage is the difference between "we shipped three codes" and "we shipped two codes and a code-review claim about a third."

NOT in `docs/IPC-SPEC.md`. Neither env var is a contract — both are test affordances. The contract is the error codes D-04 defines. Documenting either in IPC-SPEC.md would make it part of the engine's external interface, which it isn't.

Documented inline in the Swift source above each env-var read, and in the c3 commit message. Nowhere else. If either env var is ever removed (e.g., once a proper integration test exists), only those two places change.

### D-13: Audio-as-timeline reference — three coordinated mechanisms (c2 deviation from PLAN sketch)

The c2 implementation uses audio as the definitive timeline reference. Video tracks audio at every point where the two could diverge. Three mechanisms cooperate to enforce this:

**Mechanism 1 — Heartbeat dup PTS cap** (`heartbeatTick()`): When SCK stalls (primary/low-activity displays generate as few as 5fps), the heartbeat dups the last video frame but caps its PTS at `lastAppendedRawAudioPTS`. Video cannot run ahead of audio during stall periods.

**Mechanism 2 — Stop-time trim-up via `append()`** (`stop()`): After cutoffPTS is set and both pipelines drain, if `lastAppendedRawVideoPTS < lastAppendedRawAudioPTS`, a video dup is appended at `lastAppendedRawAudioPTS`. Closes the gap opened by any audio samples that arrived between the last heartbeat tick and the stop cutoff.

**Mechanism 3 — `endSession(atSourceTime:)` session bound** (`stop()`): Called with `lastAppendedRawAudioPTS - totalPausedDuration` (audio's last PTS in writer-space). This is the final safety clamp: the trim dup carries `.invalid` duration, so AVAssetWriter computes the last frame's duration using the encoder's frame interval (~33ms at 30fps). Without this bound, that implicit padding pushes `video_dur` past `audio_dur`.

#### Experimental confirmation (endSession-removal experiment, 2026-05-25)

endSession was removed (mechanisms 1 and 2 retained) and the harness run repeated over 10 takes (same setup as the c2 acceptance run: primary + external alternating, 30s takes, built-in mic).

Results:

| take | display | heartbeatDups | video_dur_s | audio_dur_s | abs_drift_ms |
|------|---------|---------------|-------------|-------------|-------------|
| 01 | primary | 95 | 29.973 | 30.016 | 42.7 |
| 02 | external | 0 | 29.928 | 30.005 | 77.0 |
| 03 | primary | 143 | 29.937 | 30.016 | 79.3 |
| 04 | external | 0 | 30.048 | 30.048 | 0.3 |
| 05 | primary | 143 | 29.935 | 30.016 | 81.0 |
| 06 | external | 0 | 29.998 | 30.027 | 28.3 |
| 07 | primary | 142 | 29.930 | 30.016 | 86.0 |
| 08 | external | 0 | 29.995 | 30.016 | 21.0 |
| **09** | **primary** | **143** | **30.115** | **30.005** | **109.7 ← FAIL** |
| 10 | external | 0 | 30.035 | 30.059 | 23.7 |

Take 9 smoking gun: 30.115s video vs 30.005s audio, 110ms drift, 143 heartbeat dups out of 150 total video frames — the primary display generated so few SCK frames that the heartbeat handled 95% of video delivery. Without endSession's session bound, AVAssetWriter added ~33ms for the trim dup's implicit last-frame duration, pushing video past audio. The c2 acceptance run (endSession present) had take 9 at 30ms drift.

Pattern: primary takes (heartbeatDups=95–143) failed at 42–110ms without endSession; external takes (heartbeatDups=0) stayed under 80ms. The failure mode is specific to high-heartbeat-dup conditions — i.e., static or low-activity displays.

**Verdict: endSession is real, not dead code. All three mechanisms stay.**

#### Why this is a semantic shift from the PLAN sketch

The PLAN characterized c2's teardown as "symmetric PTS cutoff clamp" (D-03). That framing is accurate for the stop-pipeline ordering but incomplete: the actual invariant is "audio defines the session's endpoint; video is made to match via cap + trim + bound." PLAN's D-03 covers the cutoff clamp (which mechanism 2 depends on but extends beyond), and does not anticipate the heartbeat cap (mechanism 1) or the endSession bound (mechanism 3). Both emerged from implementation.

All three mechanisms operate in writer-space against the same `totalPausedDuration` offset. Pause math is unaffected (D-02 walkthrough still holds).

### D-14: Continuity-mic teardown stall — timeout-bounded fatal-error teardown; startup latency is UX-only (c3 finding)

c3 UAT-4 (`MIC_SESSION_FAILED`, `ZEIGEN_FORCE_MIC_SESSION_ERROR=1`) passed all checks on the built-in mic. Re-running with an iPhone Continuity mic surfaced two findings via follow-up diagnostic. Neither changes D-07's bucket boundary (verified intact below).

**Finding 1 — Continuity startup latency (~3.5s), UX-only.** `AVCaptureSession.startCapture()` takes ~3.5s on first activation when the audio device is an iPhone Continuity mic. This is macOS bringing Continuity hardware online per session, not engine work. No code change. Flagged here so a future UI slice can show a "connecting to iPhone..." affordance rather than appearing hung; the engine's `ready`/`started` sequence is unchanged.

**Finding 2 — `tearDownAfterFatalError()` stall, fixed.** When a fatal error fires within ~100ms of a Continuity-backed session, `SCStream.stopCapture()` stalls indefinitely inside teardown. The writer state machine completes, but because the teardown `await` never returns, `onFatalError`'s caller never emits the IPC `error` event — the engine appears hung with no diagnostic.

Root cause is the SCK stop call, not the writer. The fix bounds only that await (`RecordingSession.tearDownAfterFatalError()`):

- The SCK stop races a 1s timeout via two detached tasks resolving a single `withCheckedContinuation`. Structured concurrency (`withTaskGroup`/`async let`) is unusable — it awaits all children even after `cancelAll()`, so an uncancellable stalled stop would never return. Detached tasks let the stalled stop be abandoned.
- 1s is chosen against the healthy-teardown profile: mid-session `stopCapture()` is sub-second under healthy conditions, well clear of 1s. The 3.5s figure from Finding 1 is *startup* — a different call — and does not apply to teardown. 1s catches the hang without truncating normal teardowns.
- On timeout, control falls through to the existing `markAsFinished` + `finishWriting` block, so the partial MP4 still lands and the `error` event still reaches IPC. The abandoned `stopCapture` is harmless: `cutoffPTS` is set at the top of teardown (before the stop), so any buffers the orphaned stop emits are clamped in `append()`; and `self.stream` is a `let` never reused (the next recording builds a fresh `RecordingSession`, `Engine.swift:192`).
- Timeout emits `TEARDOWN_TIMEOUT after <ms>ms during fatal-error path` to stderr — a breadcrumb, not an IPC contract (consistent with D-12's stderr-only stance).

Implementation note: the resume-once guard is an `OSAllocatedUnfairLock<Bool>`, not a bare `Bool`, because Swift 6 region isolation requires the flag captured by the detached (`sending`) closures to be Sendable state — a `nonisolated(unsafe) var` does not satisfy the region checker. `self` is captured directly (`RecordingSession` is `@unchecked Sendable`). `import os` added.

**D-07 verified intact — no amendment.** The diagnostic confirmed `AVCaptureSessionRuntimeError` does NOT fire on Continuity at startup or during normal operation; the Interrupted→`MIC_DROPPED` / RuntimeError→`MIC_SESSION_FAILED` bucket boundary holds on Continuity hardware. The stall in Finding 2 is a teardown-path defect downstream of the (correct) bucketing, not a misbucketed event.

### D-15: Mid-recording mic-drop robustness — cap-release + cessation watchdog (D-07 amendment; c3 UAT-1 finding)

UAT-1 (R4 mic hot-swap) surfaced two defects in D-07's mid-recording mic-drop handling. D-15 amends D-07's **coverage** — it does NOT change the Interrupted/RuntimeError bucket boundary (D-14 verified that intact). It (1) makes D-07's promised "continue video-only" actually work, and (2) adds a third drop-detection path D-07's two-notification model missed.

Mechanism was established by **code analysis**, confirmed against one real hardware take (USB-unplug on a static display). USB-unplug confirmation on an *active* display was **skipped by choice** — the freeze mechanism is deterministic in code and USB is not a target input device.

**Findings:**

1. **Heartbeat cap never releases on drop.** `heartbeatTick()` caps dup-frame PTS at `lastAppendedRawAudioPTS` unconditionally (`RecordingSession.swift:321-326`); `handleSessionInterruption` sets `micDropped` + `markAsFinished(audio)` but never resets that PTS. So after a drop, heartbeat-carried video freezes at the drop instant. D-07's "continue video-only" only ever worked for real SCK frames (active displays bypass the cap); on static/low-activity displays the heartbeat carries video and freezes. Observed: USB-unplug on a static display → `video_dur = audio_dur = 14.08s` (frozen at unplug), `regressionDropsVideo=225`.
2. **Silent sample-cessation fires no notification.** A USB device removal raises `AVCaptureDeviceWasDisconnectedNotification` (unobserved) — neither `wasInterrupted` nor `runtimeError` — so `micDropped` is never set, no `MIC_DROPPED`, no `MIC_SESSION_FAILED`. This genuine buffer-cessation (the device stops delivering) is the watchdog's real target, and it was reproduced only on USB-class hardware.

Note on iPhone Continuity — it never *ceases* at the buffer level, so it does not exercise D-15 at all. Confirmed empirically across three stimuli — screen-lock, Wi-Fi-off, and Airplane Mode (a full Wi-Fi + Bluetooth + AWDL sever): in every case audio buffers kept flowing for the full take (`audioAppended` matched the no-drop baseline; `audio_dur` = full duration). macOS keeps the AVCaptureSession audio stream alive and **substitutes silence** rather than stopping delivery. Consequence: on a Continuity departure the audio PTS keeps advancing, the heartbeat cap tracks it, and **video never freezes** — the recording degrades gracefully to a valid full-length file with a silent span, not a freeze and not an error. There is no drop to detect, so D-15(a)/(c) are inert for Continuity.

**Sub-decisions:**

- **(a) Cap-release on `micDropped` — ACCEPTED.** Two coordinated edits, both gated on `micDropped` so the normal path is untouched:
  - *a1:* `heartbeatTick()` applies the audio-PTS cap only while `!micDropped`. Once dropped, audio is finished and dup frames advance with `hostNow` — video continues video-only on any display.
  - *a2:* `stop()`'s `endSession` bound (`RecordingSession.swift:430-432`) uses `lastAppendedRawVideoPTS` when `micDropped` (video carries the timeline), else `lastAppendedRawAudioPTS` as before. Gated on `micDropped` — NOT `max(v,a)` — because in normal recording `lastVideoPTS` can sit slightly above `lastAudioPTS` from audio buffering, and clamping to audio is exactly what kills the drift D-13 mechanism 3 was added for (the take-9 110ms failure). Using `max` would reintroduce that drift.
- **(b) Observe `AVCaptureDeviceWasDisconnectedNotification` — DEFERRED to V2.3.** A faster, explicit drop signal than the watchdog, but unverified whether a full Continuity departure even fires it, and USB is not a target input. (c) closes R4 correctness as a safety net, so (b) is a V2.3 *optimization candidate* (lower drop-to-detection latency), NOT an open R4 gap.
- **(c) Mid-recording audio-cessation watchdog — ACCEPTED, 1.0s.** Track `lastAudioArrivalHost` (host clock) on every audio buffer *receipt* in `captureOutput` (before the pause/append checks, so pause doesn't false-trigger). `heartbeatTick()` (already a 0.2s timer — no new timer) checks: if audio was flowing and no buffer has arrived for >1.0s, and not paused/stopping/torn-down, route to the `MIC_DROPPED` action. Timeout rationale: healthy inter-buffer gap is ~10ms (~94 buffers/s measured) and AVCaptureAudioDataOutput buffers up to ~75ms under load; 1.0s is ~13× the worst normal gap (robust against false positives on a rare event) while detecting cessation within a second. Consistent with the D-14 teardown timeout (1.0s). 500ms was considered and rejected — false-positive safety outweighs faster detection on a rare event.

  **Validation status:** the cessation *condition* is confirmed real — the pre-fix USB take froze with `audioAppended` stuck at 1321 (buffers genuinely stopped). The post-fix positive path (watchdog fires → cap releases → video continues → audio truncates → `MIC_DROPPED reason=sample-cessation`) is verified by **code review, deliberately not hardware-fired**: neither target input produces cessation (built-in never drops; Continuity silence-substitutes per Finding 2's note), and USB is not a target input, so there is no in-scope stimulus to fire it. Re-running a USB unplug purely to exercise the path was declined by choice.

**Shared action.** `handleSessionInterruption` and the watchdog both invoke a `markMicDropped(reason:)` helper (assumes lock held): sets `micDropped`, `markAsFinished(audio)`, logs `MIC_DROPPED reason=<interrupted|sample-cessation>`. The `reason=` suffix is new (D-14 noted `AVCaptureSessionInterruptionReasonKey` is iOS-only; this gives a cause breadcrumb without it).
