## Verdict

**V2.2 complete.** The engine ships Track B (SCStream screen + per-recording AVCaptureSession mic, single AVAssetWriter, writer-start max-origin alignment, symmetric PTS-cutoff teardown). The c2 acceptance run measured `abs_drift_ms < 100ms` on 10/10 takes — well inside the V2.1-predicted `end_time_drift_ms` envelope, and substantially tighter than predicted. The V2-CHARTER DONE bar (`abs_drift_ms < 17ms`) remains V2.4's gate per R2; V2.2's bar was the envelope, not the encoder fix.

Mid-recording mic-drop robustness (D-15) and a Continuity teardown-stall fix (D-14) hardened the failure modes the new architecture introduced.

## Three-error-code coverage

All three new codes (D-04) exercised via runtime paths, not code review alone (D-12 symmetric coverage). `MIC_DROPPED` is a non-fatal stderr event, not an IPC code — included for completeness.

| code | how exercised | outcome |
|------|---------------|---------|
| `CLOCK_MISMATCH` | UAT-3, `ZEIGEN_FORCE_CLOCK_MISMATCH=1` | `error` emitted on `start`; recording never begins (0-byte output); engine stays running; re-fires on next `start`; regression-clean without the env var. |
| `MIC_NO_FIRST_SAMPLE` | c2 smoke (audio delegate disabled) | `error` ~250ms after first video sample; recording torn down; engine stays running. |
| `MIC_SESSION_FAILED` | UAT-4, `ZEIGEN_FORCE_MIC_SESSION_ERROR=1` | `error` ~100ms after `started`; partial MP4 finalized cleanly; engine returns to idle; re-fires on next `start`; regression-clean without the env var. |
| `MIC_DROPPED` (stderr, non-fatal) | UAT-1 USB-unplug (one real-hardware take) + cessation watchdog (code-verified) | Video continues video-only; audio track truncated; MP4 saves; stderr logs `MIC_DROPPED reason=<interrupted\|sample-cessation>`; no IPC event (deferred to V2.3 per D-07). |

Both fault-injection env vars (`ZEIGEN_FORCE_CLOCK_MISMATCH`, `ZEIGEN_FORCE_MIC_SESSION_ERROR`) are engineering-only test affordances — present in source with inline comments, absent from `docs/IPC-SPEC.md` and all user-facing docs (D-12).

## UAT-1 — mic hot-swap (R4)

The unified-session fear behind R4 was: a mic disappearing mid-record tears down the *whole* session, losing the recording instead of just the audio. **That catastrophic outcome does not materialize.** The recording is never lost wholesale; worst case is a valid file with a shorter or silent audio track. Two device classes behaved distinctly:

**iPhone Continuity — never ceases; degrades gracefully.** Continuity does not stop delivering audio buffers at the device level. Confirmed empirically across three departure stimuli — screen-lock, Wi-Fi-off, and Airplane Mode (a full Wi-Fi + Bluetooth + AWDL sever): in every case `audioAppended` matched the no-drop baseline and `audio_dur` ran the full take length. macOS keeps the AVCaptureSession audio stream alive and **substitutes silence** rather than stopping delivery. Consequence: audio PTS keeps advancing, the heartbeat cap tracks it, video never freezes — the file is full-length with a silent span, not a freeze and not an error. There is no drop to detect, so D-15(a)/(c) are inert for Continuity.

**USB-class — genuine cessation; D-15 fixes it.** A USB unplug raises `AVCaptureDeviceWasDisconnected` (unobserved), not Interrupted/RuntimeError, and the device stops delivering buffers. Pre-fix, this froze heartbeat-carried video at the drop instant (`video_dur = audio_dur = 14.08s`, `regressionDropsVideo=225`) and set no `micDropped`. D-15 closes both gaps:
- **(a) cap-release** — once `micDropped`, the heartbeat cap and `stop()` endSession bound release so dup frames advance with `hostNow`; video continues video-only on any display. Code-verified and confirmed against one real USB-unplug-on-static-display take.
- **(c) cessation watchdog (1.0s)** — `heartbeatTick()` (existing 0.2s timer) routes to the `MIC_DROPPED` action if no audio buffer has arrived for >1.0s while recording. The cessation *condition* is confirmed real (the pre-fix USB take froze with `audioAppended` stuck at 1321). The post-fix positive path is verified by code review, **deliberately not hardware-fired**: no target input produces cessation (built-in never drops; Continuity silence-substitutes; USB is not a target input device), so there is no in-scope stimulus to fire it.

**R4 close.** Built-in never drops; Continuity never ceases (silence-substituted graceful degradation); D-15(a) cap-release + (c) watchdog are a correctness fix plus USB-class insurance. D-15(b) (`AVCaptureDeviceWasDisconnected` observation) is deferred to V2.3 as a faster-signal *optimization* — lower drop-to-detection latency — **not an open R4 gap**, because (c) already closes correctness as a safety net. The D-07 Interrupted→`MIC_DROPPED` / RuntimeError→`MIC_SESSION_FAILED` bucket boundary held on all hardware tested (D-14 verified intact).

### D-14 — Continuity teardown-stall fix (surfaced during UAT-4 on Continuity)

Re-running UAT-4 with a Continuity mic surfaced that `SCStream.stopCapture()` stalls indefinitely inside `tearDownAfterFatalError()` when a fatal error fires within ~100ms of a Continuity-backed session — the engine appeared hung with no diagnostic. Fixed by racing the SCK stop against a 1s timeout via two detached tasks resolving one continuation (structured concurrency is unusable here — it awaits cancelled children). On timeout, control falls through to writer finalization + `error` emission, so the partial MP4 lands and IPC still gets the error. A second Continuity finding — ~3.5s session-startup latency — is UX-only (macOS bringing Continuity hardware online), no code change, flagged for a future "connecting to iPhone…" UI affordance.

## UAT-2 — clean-machine permission (R6): partially realized

**Order is code-verified.** `main.swift:16-30` pre-requests `CGRequestScreenCaptureAccess()` (screen) → `AVCaptureDevice.requestAccess(for: .audio)` (mic) → `emit(.ready)`. Both permissions gate `ready`; either denial emits `PERMISSION_DENIED` and exits non-zero. This is the contract R6 cares about, and it is satisfied.

**Live clean-machine GUI run deferred to V2.3.** A live `tccutil`-reset GUI test against the current CLI dev build would have misleading responsible-process attribution — TCC attributes screen/mic permission to the *host application* (here VS Code's integrated terminal), not a real Zeigen app bundle. The Screen Recording relaunch behavior (a granted permission not taking effect until the responsible process is relaunched) is likewise a dev-build artifact, not a property of the engine. The meaningful clean-machine permission test belongs against the **packaged Tauri app in V2.3**, where the responsible process is `com.zeigen.app`. Until then, R6 is code-verified for order; the relaunch gotcha is recorded as a known dev-build behavior to re-check at packaging.

R6 is therefore **partially realized** — order verified, GUI validation deferred — not fully closed.

## Surprises

1. c2 drift landed ~order-of-magnitude tighter than V2.1 predicted (median 3.7/16.3ms vs 62/70ms) — feeds R2.
2. The v1.0 external-worse drift asymmetry (204ms) was fully erased; post-c2 asymmetry flipped to primary-tighter — resolves c1's open question.
3. endSession bound proven load-bearing by control experiment (take 9: 30ms → 110ms when removed).
4. Continuity never ceases at the buffer level (silence-substituted) across screen-lock / Wi-Fi-off / Airplane Mode — the cessation watchdog's real target is USB-class hardware, not Continuity.
5. Continuity-backed fatal-error teardown stalled `SCStream.stopCapture()` indefinitely (D-14) — required a 1s detached-task timeout.
6. Harness `meta.csv` recorded a stale `repo_commit` (c1 `38ed647`) for a post-c2 run — a small bookkeeping nit for V2.4 to verify. (The acceptance-CSV-overwrite provenance risk is tracked as BACKLOG-V2.3-3 in `docs/PLAN.md`.)
