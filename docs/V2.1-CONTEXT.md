# V2.1: Unified-capture spike — Context

**Gathered:** 2026-05-21
**Status:** Ready for planning
**Source of truth for scope:** `docs/V2-CHARTER.md`, `docs/V2-PHASE-PLAN.md` (V2.1 section)

## Phase boundary

Standalone Swift command-line spike under `spikes/v2-unified-capture/`. Self-contained Package.swift, own build output, gitignored MP4 outputs, deletable post-merge. Imports nothing from `src-tauri/recording-engine/`; modifies nothing under it.

Goal: falsify or support the unified-clock hypothesis before V2.2 touches production code. Pass criterion is V−A drift < 17ms (single frame at 60fps) across 5 consecutive 30s recordings, on primary + at least one external display, with the user's current mic. Failure halts V2 — no productionizing a falsified hypothesis.

Out of scope for V2.1: webcam, compositing, save / edit, IPC, multiple-mic coverage, hot-swap robustness (R4), permission-flow first-run characterization beyond what's needed to make the spike run (R6), AAC priming residual full characterization (R2 — V2.4's job).

Pre-existing reality the charter undersells: **v1.0 already uses `SCStream` for both screen + mic** (via `captureMicrophone=true`, `RecordingSession.swift:153-156` and `:167-169`). The charter's "v1.0 uses a separate AVCaptureSession for mic" framing is inaccurate; the drift exists despite both sources coming from SCStream. The unified-clock hypothesis therefore concerns whether a different time base (AVCaptureSession's `masterClock` / `synchronizationClock`) produces sub-frame drift where SCStream's own time base does not.

## Carried-forward decisions

- **Done is parity, not preference** (Charter §"Done definition"): V−A drift < 1 video frame is the gate; AAC priming residual must be measured, not assumed. V2.1's tighter threshold (17ms = 60fps frame) is the spike's pass bar.
- **Spike is not load-bearing for the cutover** (Charter §"Branch hygiene"): `spikes/v2-unified-capture/` is research scaffolding; production V2 lives in `src-tauri/recording-engine/`. Deleting the spike post-V2.4 must not break anything.
- **`main` keeps shipping** (Charter §"Daily-use guarantee"): V2.1 work lands on `capture-engine-v2` only. No `main` impact.
- **Same gray-area-first discipline as Phase 10-14** (Charter §"Branch hygiene"): this CONTEXT precedes PLAN; PLAN precedes code.
- **`ffmpeg -f avfoundation` is not for screen capture on macOS 26** (CLAUDE.md §"Known gotchas"): `AVCaptureScreenInput` was removed. Whatever the V2.1 spike picks for screen is not avfoundation-CLI.
- **Use a single audio source to avoid sync drift** (CLAUDE.md §"Known gotchas"): V2.1 captures exactly one mic, no mixing.
- **Coding standards** (CLAUDE.md): simplicity, no over-engineering, no defensive programming, no emojis.

## Implementation decisions

### D-01: Screen capture path = investigate-first, two-track fallback (Gray area 1)

The spike's first job is to identify what plugs into `AVCaptureSession` on macOS 26.

- **Track A — AVCaptureSession-native screen device.** Probe macOS 26 SDK for an `AVCaptureDevice.DeviceType` representing a screen / display (candidate names: `.builtInDisplay`, `.externalScreen`, or anything returned by `AVCaptureDevice.DiscoverySession(deviceTypes: [...], mediaType: .video, position: .unspecified)` that maps to a CGDirectDisplayID). If such a device type exists → wrap with `AVCaptureDeviceInput(device:)`, add to session, share `session.masterClock` with mic input. This is the literal charter hypothesis.
- **Track B — Shared-clock fallback.** If no native screen device type exists, keep SCK for screen but extract its `synchronizationClock` (`SCStream`'s clock is reachable; CMSampleBuffers carry timestamps in that clock's time base) and set it as the AVCaptureSession's `synchronizationClock` for mic. The unified-*clock* hypothesis is satisfied; the unified-*session* implementation detail is not.
- **Track C — Falsification.** If neither track produces working code, V2.1 falsifies the v2 hypothesis. R1's fallback applies (stop V2, back out to v1.0, explore non-V2 drift mitigations).

The SPIKE-REPORT records which track delivered the result. Track A passing is a stronger result than Track B passing, but both satisfy the hypothesis as the charter actually states it ("shared master clock").

### D-02: Display arg = CGDirectDisplayID + `--list-displays` (Gray area 2)

CLI takes `--display <CGDirectDisplayID>` as an integer. `--list-displays` prints a table: `CGDirectDisplayID | width×height | is_primary | localized_name | AVCaptureDevice.uniqueID (if Track A applies)`. Resolution from CGDirectDisplayID to whatever the underlying API needs happens inside the spike.

Rationale: CGDirectDisplayID is the v1.0 convention (`RecordingSession.swift:42`, `:65-67`); the list mode is a one-line cost and saves hunting display IDs from the shell when running the same spike against primary vs external.

### D-03: Mic arg = `--mic <uniqueID>` optional, default to system default, with `--list-mics` (Gray area 3)

CLI flags:
- `--list-mics` — prints `AVCaptureDevice.uniqueID | localizedName | is_default`.
- `--mic <uniqueID>` — optional. Resolves via `AVCaptureDevice(uniqueID:)`.
- Omitted → `AVCaptureDevice.default(for: .audio)`.

Rationale: matches v1.0's `microphoneUID: String?` convention (`RecordingSession.swift:47`). Default fallback keeps the common-case invocation a single line; list mode supports the V2.4 multi-mic matrix when it arrives.

### D-04: Recording cadence = one MP4 per invocation; shell driver loops (Gray area 4)

Swift binary records exactly one MP4 per invocation and exits. A shell script at `spikes/v2-unified-capture/run.sh` drives 5 takes by calling the binary 5 times:

```sh
for i in 01 02 03 04 05; do
  ./.build/release/spike --display "$DISPLAY_ID" --mic "$MIC_UID" --duration 30 --out "output/take-${i}.mp4"
done
```

Rationale: fresh process + fresh AVCaptureSession per take is the production-shaped pattern (the engine starts once per recording). Looping inside the spike would test session-reuse behavior that doesn't exist in production. Re-running a single failing take is also a single shell command.

### D-05: Output filenames = sequential `take-NN.mp4` (Gray area 5)

`spikes/v2-unified-capture/output/take-01.mp4` through `take-05.mp4`. Driver passes `--take` / `--out` per invocation. Re-running a take clobbers the previous file — acceptable; the spike is throwaway and the SPIKE-REPORT table maps take number to drift measurement.

`spikes/*/output/` already in `.gitignore` (verified). No `.gitignore` change in this phase.

### D-06: Drift measurement = shell driver runs ffprobe (Gray area 6)

`run.sh` runs the Phase 13 ffprobe invocation after each take, appending to a CSV the SPIKE-REPORT cites:

```sh
ffprobe -v error -select_streams a:0 \
        -show_entries stream=start_time,duration \
        -of csv=p=0 "output/take-${i}.mp4"
ffprobe -v error -select_streams v:0 \
        -show_entries stream=duration \
        -of csv=p=0 "output/take-${i}.mp4"
```

Driver computes `|video_duration − audio_duration|` per take and appends to `output/results.csv` (gitignored alongside the MP4s). Swift binary stays single-purpose (record one MP4); measurement is downstream and stays in shell. If V2.1 passes, this measurement code is throwaway.

### D-07: Permissions = pre-request + document (Gray area 7)

Before constructing the AVCaptureSession, the spike pre-requests:
- `CGRequestScreenCaptureAccess()` — screen recording permission.
- `AVCaptureDevice.requestAccess(for: .audio) { granted in ... }` — mic permission.

Each gates session construction. Denial exits with a clear error code rather than letting AVCaptureSession surface its opaque "session failed to start" failure mode.

SPIKE-REPORT documents the prompt sequence observed (number of prompts, order, copy if it differs from v1.0). This double-purposes V2.1 as R6's first data point — V2.4 needs first-run characterization anyway, and the unified path may prompt differently.

### D-08: Duration = `--duration <seconds>` arg, default 30 (Pass criterion (a) literal)

Spike records for exactly the specified wall-clock duration, then stops cleanly (calls `stopRunning()` on the AVCaptureSession, `markAsFinished()` on the writer inputs, `finishWriting`). The 5-take run uses `--duration 30` per the phase plan's pass criterion. Configurable so future spike runs can validate longer durations if needed without code change.

### D-09: Exit code semantics

- `0` — recording completed, MP4 written, fwriter finished cleanly.
- Non-zero (with code on stderr) — any pre-recording failure (permission denied, display not found, mic not found, session setup failure), any mid-recording fatal (writer failure), any post-recording failure (finishWriting errored).

Driver's `for` loop should `set -e` so any failed take halts the batch. SPIKE-REPORT records partial-batch outcomes if a take fails mid-batch (R1 + R5 surface this way — a mid-batch failure on the external display is a useful data point).

## Claude's Discretion

- Exact module / file layout under `spikes/v2-unified-capture/Sources/`. Single-file is fine; a one-file spike is more honest than a structure that pretends it's production code.
- Exact CLI parser (Swift `ArgumentParser` package vs hand-rolled `CommandLine.arguments` switch). Hand-rolled is faster for ~5 flags; ArgumentParser is more ergonomic but a dependency. Lean hand-rolled.
- Exact stderr / stdout format. The driver only consumes the MP4 path and exit code; everything else is human-readable diagnostics. Plain text > JSON for the spike.
- Specific names for the spike's binary, package, and run script — keep them obvious (`spike`, `spike-v2-unified-capture`, `run.sh`).
- Whether the Track A SDK probe is its own `--probe-screen-devices` subcommand or just executes-and-prints during normal run startup. Either is fine.
- Whether the run script writes the SPIKE-REPORT.md table directly or just emits `results.csv` for me to compose into the report. Lean toward CSV + manual report composition — the table is small (5 rows × ~4 columns × 2 displays = ~40 cells) and the surprises / Track-A-vs-Track-B narrative is human-written anyway.
- The wall-clock recording length sanity check (phase plan deliverable) is `system_clock_stop − system_clock_start` measured by the driver around each invocation; spike does not need its own timer.
- Whether to capture macOS / SDK version metadata in the binary's startup log or in the driver. Lean driver (`sw_vers`, `xcrun --show-sdk-version`).
- Whether to add a `--dry-run` that exercises the SDK-probe and config path without actually recording. Optional; useful if Track A probing is itself slow or noisy.

## Deferred Ideas

- **Multi-mic coverage matrix** (built-in / USB / Continuity). Deferred to V2.4 per phase plan pass criterion (c) — "Works with the mic currently in use." Adding mic variety to V2.1 widens the spike past its hypothesis.
- **Hot-swap robustness during recording** (mic disconnect mid-take). R4 territory; surfaces in V2.4 or never. Spike does not exercise.
- **Full AAC priming residual characterization across encoder settings.** R2's full coverage is V2.4. V2.1 will produce *some* priming data as a side effect of running ffprobe, but characterizing whether priming is encoder-config-dependent (different bitrate, different sample rate, different channel count) is not the spike's job.
- **Webcam capture clock parity.** Phase plan explicitly carves webcam out — "webcam compositing stays in its own subprocess" (V2.2 sketch). The webcam ffmpeg subprocess produces a separate MP4 that compositing handles; its clock is unrelated to the V−A drift v2 is fixing.
- **IPC contract changes.** R3 governs. Spike has no IPC, so the question doesn't apply here; surfaces in V2.2.
- **Scripted SPIKE-REPORT generation.** The driver could emit a full markdown table, but the report's value is the narrative around the numbers (Track A vs B, surprises, go/no-go). Pure-data tables are cheap to compose by hand from `results.csv`.
- **Productionizing the spike's screen-capture path directly.** The spike is a research artifact (Charter §"Branch hygiene"). Even if Track A works perfectly, V2.2 builds the production engine fresh; the spike's code is not transplanted. Otherwise the spike becomes load-bearing and the charter's "deletable post-merge" guarantee breaks.
