# V2.1 — Unified-capture spike — Plan

**Drafted:** 2026-05-21
**Status:** Ready to execute
**Source of truth for decisions:** `docs/V2.1-CONTEXT.md`

A spike's deliverable is a verdict, not a feature. This plan is shaped around the SPIKE-REPORT.md output and the measurements that feed it, not around "which production file gets edited when." Production code under `src-tauri/recording-engine/` is untouched throughout.

Three commits, each a research checkpoint with its own falsifiable outcome:

- **c1 — Scaffolding + Track A attempt.** Spike binary with CLI surface (list modes, permissions, recording), exercises the AVCaptureSession-native screen device path (CONTEXT D-01 Track A). Done-when: spike either records a take via Track A OR reports a deterministic `TRACK_A_UNAVAILABLE` failure with SDK probe evidence. Both outcomes are successful c1 outcomes — the second is the data that drives c2's fallback.
- **c2 — Track B fallback + driver harness.** Adds shared-clock SCK fallback (CONTEXT D-01 Track B) to the same binary behind `--track`, plus `run.sh` driver and ffprobe measurement (CONTEXT D-04, D-06). Done-when: 5×30s × 2-display measurement matrix runs end-to-end with whichever tracks are available, results in `output/results.csv`.
- **c3 — SPIKE-REPORT.md + verdict.** Composes the report from the matrix results + Track narrative + macOS/SDK metadata + surprises. Updates `docs/V2-RISKS.md` per risk register's update protocol. Done-when: verdict is GO (Track A), GO (Track B), or NO-GO (Track C falsification) — exactly one, unambiguous.

**Falsification framing.** Track C (NO-GO) is not a failure of the spike. A confident NO-GO prevents V2.2 / V2.3 / V2.4 from being built on a false premise — that's weeks of work the spike's $0-of-production-code-touched bought. The only spike failure mode is "Maybe." If the matrix produces ambiguous results, iterate within c3 (re-run, characterize the ambiguity, decide) before committing the report.

## c1 — Scaffolding + Track A attempt

New directory: `spikes/v2-unified-capture/`. No edits to existing files.

### Files created

- `spikes/v2-unified-capture/Package.swift` — Swift 6.0 executable, macOS 15+ platform. Matches `src-tauri/recording-engine/Package.swift` shape (executable target named `spike`).
- `spikes/v2-unified-capture/Sources/spike/main.swift` — single-file CLI. Single-file is appropriate (CONTEXT Discretion); a one-file spike is more honest than a structure pretending to be production code.

### CLI surface

Hand-rolled argument parsing (CONTEXT Discretion — no ArgumentParser dependency for ~7 flags). Each flag's parse failure exits with a clear error code on stderr.

| Flag | Semantics |
|---|---|
| `--list-displays` | Prints `CGDirectDisplayID \| width×height \| is_primary \| localized_name \| AVCaptureDevice.uniqueID (or "n/a")`. Exits 0. |
| `--list-mics` | Prints `AVCaptureDevice.uniqueID \| localizedName \| is_default`. Exits 0. |
| `--display <CGDirectDisplayID>` | Required for recording. CONTEXT D-02. |
| `--mic <uniqueID>` | Optional. Default: `AVCaptureDevice.default(for: .audio)`. CONTEXT D-03. |
| `--duration <seconds>` | Default 30. CONTEXT D-08. |
| `--out <path>` | Required for recording. Driver supplies `output/<label>-take-NN.mp4`. CONTEXT D-05. |
| `--track <auto\|A\|B>` | Default `auto`. c1 supports A only — B exits `TRACK_B_NOT_IMPLEMENTED` until c2. `auto` in c1 = "try A; if A unavailable, exit TRACK_A_UNAVAILABLE" (no fallback yet). |

### Startup sequence

1. `_ = NSApplication.shared` — CG init, same pattern as `recording-engine/Sources/recording-engine/main.swift:11`.
2. Pre-request permissions (CONTEXT D-07):
   - `CGRequestScreenCaptureAccess()` for screen recording.
   - `AVCaptureDevice.requestAccess(for: .audio) { granted in ... }` for mic — block on the callback.
   - If either denied: exit `PERMISSION_DENIED` with which permission failed on stderr.
3. Resolve `--display` against `CGGetActiveDisplayList`. Not found → exit `DISPLAY_NOT_FOUND`.
4. Resolve `--mic` via `AVCaptureDevice(uniqueID:)` if passed; otherwise `AVCaptureDevice.default(for: .audio)`. Explicit UID that doesn't resolve → exit `MIC_NOT_FOUND`.
5. Dispatch on `--track`.

### Track A implementation

SDK probe for a screen `AVCaptureDevice.DeviceType` on macOS 26:

1. Try `AVCaptureDevice.DiscoverySession(deviceTypes: [<candidate>], mediaType: .video, position: .unspecified)` with each candidate that exists in the SDK enum: `.builtInDisplay`, `.externalScreen`, anything else that smells screen-shaped. Use `#available` / conditional compilation so the spike builds even if the enum case doesn't exist.
2. Fall back to enumerating `AVCaptureDevice.devices(for: .video)` and string-matching `deviceType.rawValue` for "display" / "screen."

If a screen device is identified:
- Construct `AVCaptureDeviceInput(device: screenDevice)`, add to a fresh `AVCaptureSession`.
- Construct `AVCaptureDeviceInput(device: micDevice)`, add to the same session.
- Verify both inputs share `session.synchronizationClock` (macOS 14+; `masterClock` is the older spelling — use whichever the current SDK exposes).
- Add `AVCaptureMovieFileOutput`. MovieFileOutput is simpler for the spike than wiring `AVCaptureVideoDataOutput` + `AVAssetWriter` by hand — under the hood it's still an AVAssetWriter, and the measurement only cares about the resulting MP4. (Production V2.2 may want AVAssetWriter for finer control — that decision belongs in V2.2 CONTEXT, not here.)
- `session.startRunning()`, sleep `--duration` seconds (in a way that doesn't block the AVCaptureSession queue — use `DispatchSemaphore` or run-loop drain), `output.stopRecording(to: URL, recordingDelegate: ...)`, await delegate completion, `session.stopRunning()`.
- Write `<out>.track` sidecar containing `A` (so the driver can record which path produced the file — keeps stdout/stderr free for diagnostics).
- Exit 0.

If no screen device type is found:
- stderr: `TRACK_A_UNAVAILABLE: no AVCaptureDevice.DeviceType for screens found on macOS <version>, SDK <version>. Enumerated device types: [<list>]`.
- Exit `TRACK_A_UNAVAILABLE`.

This is a *finding*, not a bug. In c2, `--track auto` will read this exit code and fall through to B.

### c1 done-when

- `cd spikes/v2-unified-capture && swift build -c release` succeeds; binary at `.build/release/spike`.
- `./spike --list-displays` and `./spike --list-mics` produce sensible tables (eyeball — at least one display, at least one mic, default mic flagged).
- `./spike --display $PRIMARY_ID --duration 5 --out output/smoke.mp4 --track A` produces exactly one of:
  - **(A-success)** A playable MP4 at `output/smoke.mp4` with both video + audio streams (verify via `ffprobe -show_streams`). Sidecar `output/smoke.mp4.track` contains `A`. Exit 0.
  - **(A-unavailable)** Exit `TRACK_A_UNAVAILABLE` with stderr listing enumerated device types. No MP4 written. **This is a successful c1 outcome** — it is the SDK-probe data c2 and c3 depend on.
- (A-success only) Log `session.synchronizationClock` identity to stderr at `startRunning` time and confirm both inputs reference the same clock object. If the screen input and mic input don't share a clock, that's a silent Track A failure even with a playable MP4 — flag it and downgrade to `TRACK_A_CLOCK_MISMATCH` exit.
- Permission failure path exercised once: revoke Screen Recording for the spike binary in System Settings → Privacy & Security, re-run, verify `PERMISSION_DENIED` exits cleanly (not the opaque AVCaptureSession "session failed" surface). Re-grant after.

## c2 — Track B fallback + driver harness

Adds Track B to the same `main.swift`. Adds `run.sh` driver. No production code touched.

### Track B implementation

`--track B` (and `--track auto` when Track A returns `TRACK_A_UNAVAILABLE`):

- Construct `SCStream` for the display following v1.0 RecordingSession `:65-73` patterns (SCDisplay lookup, SCContentFilter, SCStreamConfiguration). **Do not** set `captureMicrophone=true` — Track B routes mic through AVCaptureSession deliberately.
- Construct `AVCaptureSession` with mic input (`AVCaptureDeviceInput(device: micDevice)`) only. Add `AVCaptureAudioDataOutput` with a delegate — we need raw CMSampleBuffers to thread into the writer, so `AVCaptureMovieFileOutput` won't work here.
- Investigate SCStream's clock surface. SCStream's CMSampleBuffers carry presentation timestamps in some clock's time base — identify whether it's `CMClockGetHostTimeClock()` (likely on macOS) or an SCK-internal clock reachable via SCStream API. **Set it as the AVCaptureSession's `synchronizationClock` (macOS 14+).** If the SCStream clock is `CMClockGetHostTimeClock()` and that's also AVCaptureSession's default master clock, the assignment may be a no-op confirming both already share host time — that's still the success condition for shared-clock parity.
- Construct an `AVAssetWriter` (mp4) with video + audio inputs. Reuse v1.0 RecordingSession settings (`:111-143` — H.264 8Mbps for video, AAC 48kHz mono 128kbps for audio) so SPIKE-REPORT measurements are apples-to-apples vs v1.0 baselines. CONTEXT §"Reusable references" lists the exact lines.
- Feed SCStream screen output → video `AVAssetWriterInput`.
- Feed AVCaptureSession audio output (via `AVCaptureAudioDataOutput` delegate) → audio `AVAssetWriterInput`.
- Both inputs' CMSampleBuffer timestamps now share a single clock — writer session time base is well-defined.
- Run for `--duration` seconds. Teardown order: stop SCStream → stop AVCaptureSession → `markAsFinished` on both writer inputs → `finishWriting`.
- Write `<out>.track` sidecar containing `B`. Exit 0.

If anything in the SCStream-or-AVCaptureSession setup fails before the recording starts (e.g. shared-clock assignment errors), exit `TRACK_B_SETUP_FAILED` with the underlying error on stderr. Track B genuinely failing-to-build is a Track C signal — the shared-clock path doesn't compile / doesn't run on this SDK at all.

### Driver harness

`spikes/v2-unified-capture/run.sh`:

```sh
#!/bin/bash
set -e

PRIMARY="${1:?usage: run.sh <primary_display_id> <external_display_id> [mic_uid] [track]}"
EXTERNAL="${2:?missing external display id}"
MIC_UID="${3:-}"
TRACK_OVERRIDE="${4:-auto}"   # auto | A | B — defaults to spike's auto-select

SPIKE="./.build/release/spike"
OUT="output"
mkdir -p "$OUT"

# Metadata snapshot.
{
  echo "macos_version,$(sw_vers -productVersion)"
  echo "sdk_version,$(xcrun --show-sdk-version --sdk macosx 2>/dev/null || echo unknown)"
  echo "track_override,$TRACK_OVERRIDE"
  echo "primary_display,$PRIMARY"
  echo "external_display,$EXTERNAL"
  echo "mic_uid,${MIC_UID:-default}"
} > "$OUT/meta.csv"

echo "display,take,track,wall_clock_s,video_dur_s,audio_start_s,audio_dur_s,abs_drift_ms,exit_code" \
  > "$OUT/results.csv"

for LABEL in primary external; do
  case "$LABEL" in
    primary)  DISPLAY_ID="$PRIMARY"  ;;
    external) DISPLAY_ID="$EXTERNAL" ;;
  esac
  for i in 01 02 03 04 05; do
    OUTFILE="$OUT/${LABEL}-take-${i}.mp4"
    rm -f "$OUTFILE" "$OUTFILE.track"

    START=$(date +%s.%N)
    set +e
    "$SPIKE" --display "$DISPLAY_ID" \
             ${MIC_UID:+--mic "$MIC_UID"} \
             --duration 30 \
             --track "$TRACK_OVERRIDE" \
             --out "$OUTFILE"
    EXIT=$?
    set -e
    END=$(date +%s.%N)
    WALL=$(echo "$END - $START" | bc)

    if [ "$EXIT" -eq 0 ] && [ -f "$OUTFILE" ]; then
      TRACK=$(cat "$OUTFILE.track" 2>/dev/null || echo "?")
      VDUR=$(ffprobe -v error -select_streams v:0 -show_entries stream=duration -of csv=p=0 "$OUTFILE")
      ASTART=$(ffprobe -v error -select_streams a:0 -show_entries stream=start_time -of csv=p=0 "$OUTFILE")
      ADUR=$(ffprobe -v error -select_streams a:0 -show_entries stream=duration -of csv=p=0 "$OUTFILE")
      DRIFT_MS=$(echo "($VDUR - $ADUR) * 1000" | bc | awk '{print ($1 < 0 ? -$1 : $1)}')
    else
      TRACK="n/a"; VDUR=""; ASTART=""; ADUR=""; DRIFT_MS=""
    fi
    echo "$LABEL,$i,$TRACK,$WALL,$VDUR,$ASTART,$ADUR,$DRIFT_MS,$EXIT" >> "$OUT/results.csv"
  done
done

echo
echo "Results in $OUT/results.csv:"
column -t -s, "$OUT/results.csv"
```

Single take re-run: `./.build/release/spike --display X --duration 30 --out output/replay.mp4 --track B` works independently of the matrix.

### c2 done-when (matrix runs)

- `swift build -c release` succeeds with both tracks compiled in.
- `./run.sh $PRIMARY $EXTERNAL` runs all 10 invocations to completion. "Completion" means: each spike invocation exits (0 on success, non-zero with a clear code on failure); `results.csv` has 10 rows. A row with `exit_code != 0` is a recorded outcome, not a c2 blocker.
- For at least one track (A or B), at least one display produces 5 takes with `abs_drift_ms` populated. If neither track produces *any* successful take on *any* display, that is itself a Track C signal — proceed to c3 with the falsification narrative rather than iterating on c2.
- Single-take re-run works for both `--track A` (if A available) and `--track B` independently of `run.sh`.

### Track-specific pass bars (evaluated at c3 verdict time)

These are not c2 done-when criteria — they are the verdict criteria c3 applies to the matrix `results.csv`. Listed here so the implementation knows what numbers it's hunting:

- **Track A passes** iff all 5 takes on both displays via `--track A` exit 0 and have `abs_drift_ms < 17` on every row. (10 of 10 rows under threshold.)
- **Track B passes** iff all 5 takes on both displays via `--track B` exit 0 and have `abs_drift_ms < 17` on every row. (10 of 10 rows under threshold.)
- **Partial** = some takes pass, others don't, or one display passes and the other doesn't. Partial does not satisfy the spike's pass bar, but it is a documented finding R5 (multi-display) cares about — feeds the c3 narrative and may inform V2.2's display-handling design.
- If both tracks are partial or worse → Track C (NO-GO) verdict in c3.

## c3 — SPIKE-REPORT.md + verdict

`spikes/v2-unified-capture/SPIKE-REPORT.md` — the phase deliverable per V2-PHASE-PLAN.md §"V2.1 Deliverable."

### Execution before composing the report

1. Run `./run.sh $PRIMARY $EXTERNAL` end-to-end. Confirm `output/results.csv` has 10 rows.
2. If both Track A and Track B are available, run the matrix twice more — `./run.sh $PRIMARY $EXTERNAL "" A` and `./run.sh $PRIMARY $EXTERNAL "" B` — so the report can compare both paths under controlled conditions. (The default `auto` run reflects production-shape behavior, which is what V2.2 will productionize.)
3. If the auto-run results are clean and unambiguous (all-pass or all-fail for whichever track auto picked), step 2's controlled re-runs are optional — proceed to compose the report.

### Report contents (per V2-PHASE-PLAN.md V2.1 Deliverable)

- **macOS + SDK metadata** — populated from `output/meta.csv`.
- **Display IDs and mic UID used** — copied from the spike's `--list-displays` / `--list-mics` outputs run pre-matrix.
- **Track availability narrative** — which tracks ran, which were unavailable. If Track A is unavailable, paste the full `TRACK_A_UNAVAILABLE` stderr block including enumerated device types. This is the data future V2-class work will look back on if anyone asks "did we check Track A?"
- **Results table** — one row per take across however many tracks ran. Columns from `results.csv`: display | take | track | wall_clock_s | video_dur_s | audio_start_s | audio_dur_s | abs_drift_ms | exit_code.
- **AAC priming residual snapshot** — one `audio_start_s` data point per track. Note in prose whether the value is near 0 (priming fixed by the unified path) or near 0.043 (priming matches v1.0 behavior — R2's pre-stated possible outcome). One data point is enough for V2.1; R2's full characterization is V2.4's job.
- **Surprises** — anything unexpected during construction or execution. Each surprise either edits `docs/V2-RISKS.md` (existing risk gets new data) or adds a new risk (per risk register update protocol).
- **Verdict** — exactly one line, exactly one of:
  - **GO (Track A).** All 10 Track A takes < 17ms. V2.2 productionizes the AVCaptureSession-native screen device path.
  - **GO (Track B).** All 10 Track B takes < 17ms (and either Track A unavailable or Track A failed). V2.2 productionizes the shared-clock SCK fallback.
  - **NO-GO (Track C — falsification).** Both tracks fail the 17ms bar on at least one take, OR both tracks fail to set up at all. The shared-clock hypothesis does not hold in software at the granularity v2 requires. R1's fallback applies: V2 stops, v1.0 remains daily driver, drift mitigation reverts to downstream compensation in a new (non-V2) phase.

### V2-RISKS.md updates in the same commit

Per V2-RISKS.md §"Update protocol" — a spike finding edits the risk register in the same commit as the report.

- **R1** moves from "spike coverage" to one of "realized — GO via Track A", "realized — GO via Track B", or "realized — NO-GO; v2 hypothesis falsified; fallback engaged."
- **R2** gains a measured data point on AAC priming residual under the unified path. Severity unchanged.
- **R5** gains data if the matrix showed per-display differences (one display passed, another didn't). If yes, note it as a constraint V2.2's screen-handling needs to address.
- **R6** gains data on the permission prompt sequence observed during c1's first run. If the unified path prompts differently from v1.0's separate-permission flow, note it — it's V2.4's first-run-parity concern.

### c3 done-when

- `spikes/v2-unified-capture/SPIKE-REPORT.md` exists, populated from real measurements (no placeholders).
- `docs/V2-RISKS.md` updated in the same commit, R1's status moved to realized.
- Verdict line is GO (Track A), GO (Track B), or NO-GO. **No "TBD," no "needs more data."** If more data is needed, the spike isn't done — re-run, characterize, and only commit when the verdict is unambiguous.
