# V2.2 — Productionize unified capture — Plan

**Drafted:** 2026-05-23
**Status:** Ready to execute
**Source of truth for decisions:** `docs/V2.2-CONTEXT.md`

V2.2's deliverable is an engine binary that ships Track B's architecture against a measurable bar (D-11: `abs_drift_ms < 100ms` on 10/10 takes), plus a harness V2.4 reuses, plus failure-mode instrumentation around the new architecture. Three commits, each atomic and each independently verifiable:

- **c1 — Standalone drift harness.** Build `harness/v2-drift/` with the runner that drives the engine binary via stdin/stdout JSON IPC and measures drift via ffprobe. Run against unmodified v1.0 to confirm the harness methodology produces sensible numbers (~Phase 13 order of magnitude). No engine changes.
- **c2 — Engine rewrite.** Rewrite `RecordingSession.swift` to Track B shape with writer-start alignment, symmetric teardown, two sample handlers, and `MIC_NO_FIRST_SAMPLE` timeout. Smallest path-to-bar commit; the only new IPC error code introduced here is `MIC_NO_FIRST_SAMPLE`. Acceptance: c1's harness against new engine, 10/10 takes `abs_drift_ms < 100ms`.
- **c3 — Robustness + V2.2 acceptance UATs.** Add `CLOCK_MISMATCH` verification (per-`start`), `MIC_SESSION_FAILED` runtime observer, `MIC_DROPPED` interruption handler, permission pre-request at engine startup, `ZEIGEN_FORCE_CLOCK_MISMATCH` env-var fault injection. Run hot-swap and clean-machine permission UATs. Compose `docs/V2.2-REPORT.md`; update `docs/V2-RISKS.md` (R4, R6).

**Atomicity guarantee.** c1 ships independently of the rewrite (harness against v1.0). c2 ships an engine that hits the drift bar but is missing failure-mode instrumentation (intermediate state acceptable on `capture-engine-v2`; branch does not ship to users). c3 closes the failure-mode gap and produces the phase report.

**Out-of-scope reminder** (V2-CHARTER): no UI changes, no save/edit pipeline changes, no webcam touches, no Tauri-Rust wiring (V2.3), no R2 priming fix (V2.4), no architectural cleanups beyond what Track B requires (D-09 limiter callout — move the call site, do not touch the behavior).

## c1 — Standalone drift harness

New directory: `harness/v2-drift/`. No edits to existing engine code. No production code touched.

### Files created

- `harness/v2-drift/run.py` — Python driver that spawns the engine binary, drives it via line-delimited JSON over stdin/stdout, gathers 10 MP4s and per-take measurements into CSV.
- `harness/v2-drift/output/.gitkeep` — keeps the directory present in repo (output `.mp4`, `.csv` files themselves are gitignored).

### Files edited

- `.gitignore` — add `harness/v2-drift/output/*.mp4` and `harness/v2-drift/output/*.csv`. Keep `output/.gitkeep` tracked.

### Driver shape

Python (CONTEXT D-10 Discretion: shell + Python; consolidating into pure Python is simpler than mixing). Standard library only — no pip deps.

```text
harness/v2-drift/run.py [--engine <path>] [--takes 10] [--duration 30]
                        [--displays primary,external]
```

Default `--engine` resolves to `../../src-tauri/recording-engine/.build/release/recording-engine` relative to the script. Script behavior:

1. Build the engine binary if missing (`swift build -c release` in the engine directory).
2. Spawn the engine subprocess; pipe stdin/stdout for JSON, stderr to harness's stderr.
3. Wait for `{"event":"ready",...}`.
4. Send `{"command":"enumerate"}`; parse `enumerated` event; capture display IDs and pick (primary, one external).
5. For each take in 1..N:
   - `output_path = output/take-<label>-<NN>.mp4`
   - Send `{"command":"start", "display_id": <id>, "microphone_uid": <default mic uid>, "output_path": ..., "max_fps": 30}`.
   - Wait for `started`.
   - Sleep `--duration` seconds (poll for `progress` events as a liveness signal; if no `progress` for 5s, treat as engine hang and abort take).
   - Send `{"command":"stop"}`.
   - Wait for `stopped`.
6. After all takes, send `{"command":"quit"}`; reap the subprocess.
7. Run ffprobe on each MP4:
   - `ffprobe -v error -select_streams v:0 -show_entries stream=duration -of csv=p=0 <mp4>` → `video_duration`.
   - `ffprobe -v error -select_streams a:0 -show_entries stream=start_time,duration -of csv=p=0 <mp4>` → `audio_start_time`, `audio_duration`.
8. Compute `abs_drift_ms = abs(video_duration - audio_duration) * 1000`.
9. Write `output/results.csv` with columns: `display,take,wall_clock_s,video_dur_s,audio_start_s,audio_dur_s,abs_drift_ms,exit_code,output_path`.
10. Write `output/meta.csv` with: macOS version (`sw_vers -productVersion`), SDK version (`xcrun --show-sdk-version --sdk macosx`), engine commit hash (`git -C <engine-dir> rev-parse HEAD`), harness commit hash, default mic UID, displays used.

### Engine-IPC dance details

- One engine subprocess per harness invocation; 10 takes drive through that single process (matches production daily-use pattern: one engine per UI session, multiple recordings). Spike's "fresh process per take" pattern is not the production shape.
- Read engine stdout one JSON line at a time (`stdout.readline()` → `json.loads`). Time out reads at 60s; on timeout, log and kill subprocess.
- Engine stderr lines forwarded to harness stderr (prefixed `engine: `) for diagnostic visibility.

### Display variation

Default behavior alternates display per take: takes 1, 3, 5, 7, 9 on primary; 2, 4, 6, 8, 10 on external. Matches the V2.1 spike's 5×2 matrix shape but in one CSV without re-invoking the script.

If only one display is present (`enumerated` returns one), run all 10 on that display; record the constraint in `meta.csv`.

### Rewrite shape — what changes inside RecordingSession.swift

Removed:
- `config.captureMicrophone = true` and `config.microphoneCaptureDeviceID` from SCStreamConfiguration (lines 153-156).
- `stream.addStreamOutput(self, type: .microphone, ...)` (lines 167-169).
- `.microphone` branch in `stream(_:didOutputSampleBuffer:of:)` (lines 283-286).

Added:
- A per-recording `AVCaptureSession` member, constructed in `init` (after the SCStream construction so the failure modes are ordered: SCStream-setup failure → no AVCaptureSession leaks).
- `AVCaptureDeviceInput(device: micDevice)` + `AVCaptureAudioDataOutput` with audio settings forcing mono float PCM 48kHz (the spike's exact settings, `Sources/spike/main.swift:377-385` — matches the Phase 12 limiter's expected `UnsafeMutablePointer<Float>` input shape, which is the verification step from CONTEXT FLAG 2).
- A new sample-buffer-delegate method `captureOutput(_:didOutput:from:)` implementing `AVCaptureAudioDataOutputSampleBufferDelegate`. Body: validate buffer, apply limiter in place (preserving D-09 behavior — same `applyLimiterInPlace` call, same Phase 12 limiter, just at a new call site), call `append(buffer, to: audioInput, isVideo: false)`.
- `firstVideoPTS` / `firstAudioPTS` state members (`CMTime`, initially `.invalid`).
- A `writerStarted: Bool` computed property reading `started` under lock — exposed for Engine.swift's `handlePause` check (CONTEXT D-06 mechanism).
- `cutoffPTS: CMTime?` state member, set at `stop()` entry.

Modified `append(_, to:, isVideo:)` (lines 294-346) — replaces existing implementation. New flow inside the lock:

1. If `paused`, drop. (Unchanged from v1.0.)
2. **Read pipeline's first PTS** — if this is the first sample for video or audio, record `firstVideoPTS` or `firstAudioPTS`.
3. **Writer-start gate (NEW)** — if `!started`:
   - If both `firstVideoPTS.isValid` AND `firstAudioPTS.isValid`: compute `origin = max(firstVideoPTS, firstAudioPTS)`; call `writer.startWriting()` + `writer.startSession(atSourceTime: origin)`; set `sessionStartPTS = origin`; set `started = true`. Continue to step 4.
   - Else: this sample is from the only-warmed-up pipeline; drop it. Unlock and return. The `MIC_NO_FIRST_SAMPLE` timer is a separate concern (see below).
4. **PTS cutoff clamp (NEW)** — if `cutoffPTS` set and `rawPTS > cutoffPTS`: drop. Unlock and return.
5. **Pre-origin drop (NEW)** — if `rawPTS < sessionStartPTS`: drop. (Belt-and-suspenders for the case where the earlier pipeline's leftover queue produces a sample slightly after writer-start but with a pre-origin PTS. CMSampleBuffer queues can be reordered.)
6. **Pause offset** — `adjustedPTS = rawPTS - totalPausedDuration`. (Unchanged from v1.0.)
7. **Append** — same as v1.0: `input.isReadyForMoreMediaData` check, then `input.append(finalBuffer)`, with counter bookkeeping.

### MIC_NO_FIRST_SAMPLE timeout mechanism

After `start()` returns and the recording is running, schedule a one-shot timer 250ms after the first video sample arrives. If at timer fire `firstAudioPTS.isValid == false`, throw `EngineError(code: "MIC_NO_FIRST_SAMPLE", message: ...)` to Engine.swift via the error-callback (see "Mid-recording error emission" below).

Implementation: in `append()`'s first-video-sample branch (after writing `firstVideoPTS`), kick off:

```swift
DispatchQueue.global().asyncAfter(deadline: .now() + 0.250) { [weak self] in
    self?.checkMicTimeout()
}
```

`checkMicTimeout()` acquires the lock, checks `firstAudioPTS.isValid`. If false, signals failure to Engine via a callback. If true, no-op. Single-shot; no rescheduling.

The "error callback" is a closure passed to RecordingSession's constructor by Engine.swift — see next section.

### Mid-recording error emission path

Engine.swift's `handleStart()` constructs `RecordingSession` and stores it. Mid-recording errors (MIC_NO_FIRST_SAMPLE in c2; MIC_SESSION_FAILED + MIC_DROPPED in c3) need to surface back to Engine for IPC emission + state reset.

Mechanism: `RecordingSession.init` takes an additional parameter `onFatalError: (EngineError) -> Void` (or `@Sendable (EngineError) -> Void` for actor compatibility). RecordingSession invokes the callback when an unrecoverable mid-recording condition occurs. Engine.swift constructs the callback as:

```swift
let onFatalError: (EngineError) -> Void = { [weak self] err in
    Task { [weak self] in
        await self?.handleFatalError(err)
    }
}
```

`handleFatalError(_:)` is an actor-isolated method that:
1. Emits `.error(code: err.code, message: err.message)`.
2. Calls `session?.tearDownAfterFatalError()` — finishes whatever has been written, releases SCStream + AVCaptureSession.
3. Sets `self.session = nil` and `self.state = .idle`.

`tearDownAfterFatalError()` is the symmetric-teardown path (D-03) but invoked from the error context instead of from `stop()`.

c2 only wires `MIC_NO_FIRST_SAMPLE` through this callback. c3 adds the AVCaptureSession runtime-error observer that also routes through it.

### Verification steps in c2

These are PLAN-level verifications, not done-when criteria:

1. **Limiter format check (FLAG 2).** Confirm the AVCaptureAudioDataOutput audio settings (mono Float32 PCM 48kHz) match the limiter's expected input shape. `applyLimiterInPlace` reads `AudioBufferList` and casts `mData` to `UnsafeMutablePointer<Float>`. Mono Float32 PCM 48kHz produces a single buffer of `Float`-typed samples — matches the cast. If the AVCaptureAudioDataOutput honors the `audioSettings` request, the format is correct. Smoke test: log the first audio sample's `AudioStreamBasicDescription` to stderr on first sample arrival; verify `mFormatID == kAudioFormatLinearPCM`, `mBitsPerChannel == 32`, `mChannelsPerFrame == 1`, `mSampleRate == 48000`, format flag includes `kAudioFormatFlagIsFloat`. Remove the smoke log before commit.

2. **Heartbeat carry-forward check (FLAG 3).** The v1.0 heartbeat (`RecordingSession.swift:177-216`) duplicates the last video buffer every 200ms if SCK stalls. Two interactions with c2:
   - Heartbeat doesn't fire until first video sample has arrived (`lastVideoBuffer` is nil until then). So heartbeat cannot fire before writer-start gate (which waits for both pipelines).
   - Heartbeat injects a video sample via `append()` — the same `append()` that respects the cutoff clamp. Stop teardown sets cutoff, heartbeat cancels timer; race window is ~ms-scale; cutoff clamp catches any in-flight heartbeat sample.

   Conclusion: heartbeat carries forward as-is. No changes needed. PLAN flags this as reviewed, not as a code change.

3. **Pause math walkthrough.** From CONTEXT D-02 ("PLAN includes an explicit walkthrough"):
   - Recording starts. Video first PTS = `t_v`; audio first PTS = `t_a`; `t_v < t_a` typically.
   - Writer-start fires at `origin = max(t_v, t_a) = t_a`. `sessionStartPTS = t_a`.
   - First few video samples between `t_v` and `t_a` were pre-origin → dropped in `append()` (step 5).
   - At `t = t_a`, output timeline = 0.
   - Recording runs. At `t = t_a + 10s`, user pauses. `paused = true`, `pauseStartedAt = t_a + 10s`.
   - Samples between `t_a + 10s` and resume are dropped in `append()` step 1.
   - At `t = t_a + 12s`, user resumes. `paused = false`, `totalPausedDuration += 2s`.
   - Sample at `rawPTS = t_a + 12s` arrives. `adjustedPTS = (t_a + 12s) - 2s = t_a + 10s`. Output-timeline duration = `adjustedPTS - sessionStartPTS = 10s`. Continuous from pre-pause output timeline.
   - On stop at `t_a + 20s`, output timeline = 18s (20s wall - 2s paused). MP4 plays gaplessly across the pause point.

   Math holds because both pipelines share host time and both honor the same `totalPausedDuration` offset.

### CLOCK_MISMATCH verification — implementation

After `stream.startCapture()` and `session.startRunning()` complete in `RecordingSession.start()`:

```swift
private func verifyClockParity() throws {
    if ProcessInfo.processInfo.environment["ZEIGEN_FORCE_CLOCK_MISMATCH"] == "1" {
        throw EngineError(code: "CLOCK_MISMATCH", message: "forced mismatch via ZEIGEN_FORCE_CLOCK_MISMATCH=1")
    }
    guard let sessionClock = session.synchronizationClock else {
        throw EngineError(code: "CLOCK_MISMATCH", message: "AVCaptureSession.synchronizationClock is nil after startRunning")
    }
    let hostNow1 = CMClockGetTime(CMClockGetHostTimeClock())
    let sessionUptime = CMClockConvertHostTimeToSystemUptime(hostNow1)
    let hostNow2 = CMClockGetTime(CMClockGetHostTimeClock())
    let delta = CMTimeGetSeconds(hostNow2 - hostNow1)
    if delta > 0.001 || sessionUptime.timeValue == 0 {
        throw EngineError(code: "CLOCK_MISMATCH", message: "host-time round-trip drift \(delta)s exceeds 1ms tolerance")
    }
}
```

The round-trip check is loose by design — we're confirming the conversion succeeds and host time is making progress, not measuring sub-microsecond precision. V2.1 already proved time-base parity; this is a runtime smoke that the SDK still cooperates.

`verifyClockParity()` is called inside `start()` after `session.startRunning()`. On throw, `start()` propagates the EngineError up to Engine.swift's `handleStart`, which emits via the existing `catch EngineError` path.

### AVCaptureSession notification observer — implementation

In `RecordingSession.init()`, after constructing the AVCaptureSession, register two observers (NotificationCenter, not actor-isolated — observer closures dispatch to a serial queue that calls into RecordingSession's lock-protected methods):

```swift
NotificationCenter.default.addObserver(
    forName: .AVCaptureSessionWasInterrupted,
    object: session,
    queue: nil
) { [weak self] note in
    self?.handleSessionInterruption(note)
}
NotificationCenter.default.addObserver(
    forName: .AVCaptureSessionRuntimeError,
    object: session,
    queue: nil
) { [weak self] note in
    self?.handleSessionRuntimeError(note)
}
```

`handleSessionInterruption` (CONTEXT D-07 bucket 1, MIC_DROPPED):
1. Acquire lock; if already torn down (`micDropped` flag set), return.
2. Set `micDropped = true`.
3. Read interruption reason from `note.userInfo[AVCaptureSessionInterruptionReasonKey]`; log `MIC_DROPPED reason=<reason>` to stderr via `logStderr`.
4. Call `audioInput.markAsFinished()`.
5. Release lock. Recording continues with video only. No IPC event (D-07 explicit).

`handleSessionRuntimeError` (CONTEXT D-07 bucket 2, MIC_SESSION_FAILED):
1. Acquire lock; if already torn down, return.
2. Extract `AVError` from `note.userInfo[AVCaptureSessionErrorKey]`; build message.
3. Release lock.
4. Invoke `onFatalError(EngineError(code: "MIC_SESSION_FAILED", message: ...))`. Engine.swift's `handleFatalError` (added in c2) emits + tears down.

In `append()`'s audio branch: if `micDropped`, drop the sample silently. (Edge case: late-arriving audio sample after `audioInput.markAsFinished()`. AVAssetWriterInput rejects writes after `markAsFinished`; the early drop is hygienic, not strictly required.)

Remove observers in RecordingSession's `deinit` / teardown path.

### MIC_SESSION_FAILED fault injection — implementation

Per CONTEXT D-12, the synthetic fault path. After `session.startRunning()` returns and `verifyClockParity()` passes, check `ZEIGEN_FORCE_MIC_SESSION_ERROR`:

```swift
if ProcessInfo.processInfo.environment["ZEIGEN_FORCE_MIC_SESSION_ERROR"] == "1" {
    DispatchQueue.global().asyncAfter(deadline: .now() + 0.100) { [weak self] in
        guard let self else { return }
        let synthetic = AVError(.deviceNotConnected, userInfo: [
            NSLocalizedDescriptionKey: "synthetic mic session error via ZEIGEN_FORCE_MIC_SESSION_ERROR=1"
        ])
        self.onFatalError(EngineError(
            code: "MIC_SESSION_FAILED",
            message: "synthetic: \(synthetic.localizedDescription)"
        ))
    }
}
```

The 100ms delay simulates a runtime-error notification firing a beat after capture has started — matches the timing window real runtime errors would fire in. The `AVError` payload mirrors what a real `handleSessionRuntimeError` would extract, so the downstream `onFatalError` → `handleFatalError` → emit + teardown path is exercised identically.

Comment directly above the env-var read calls out the engineering-only nature per CONTEXT D-12.

### Permission pre-request — implementation

In `main.swift`, before `emit(.ready(version: "0.1.0"))`:

```swift
let screenGranted = CGRequestScreenCaptureAccess()
if !screenGranted {
    emit(.error(code: "PERMISSION_DENIED", message: "Screen Recording permission not granted"))
    exit(1)
}

let micGranted = await withCheckedContinuation { (c: CheckedContinuation<Bool, Never>) in
    AVCaptureDevice.requestAccess(for: .audio) { ok in c.resume(returning: ok) }
}
if !micGranted {
    emit(.error(code: "PERMISSION_DENIED", message: "Microphone permission not granted"))
    exit(1)
}

emit(.ready(version: "0.1.0"))
```

Same shape as the spike's pre-request (`spikes/v2-unified-capture/Sources/spike/main.swift:166-175`). The `await` requires `main.swift` to be in an async context — current `main.swift` uses `while let line = readLine() { ... await engine.handle(cmd) }`, so adding an `await` for permission is supported.

Camera permission is NOT pre-requested. Per CONTEXT D-08.
