## N-01 — Parity-gate measurement target: measure engine output, not the saved webcam composite

**Captured:** 2026-05-28 (V2.3, between c2 and c3)
**Source:** a one-clip end-to-end sync probe pulled forward from V2.4's parity
matrix — a real recording driven through the full UI (countdown, screen +
built-in mic + webcam, ~25.6s, stop, **Save**), then ffprobed at every stage.
Clip: `recording-2026-05-28-081925`.

### Finding

Engine A/V sync survives end-to-end. The drift that shows up on the **saved**
file is the compositor's **intentional** webcam-lag compensation (a deliberate
duration de-match for perceptual alignment), not a sync regression. The clip
"looked synced" on playback, consistent with the compensation doing its job.

### Stage-localized drift (same clip — controls for take-to-take variance)

Metric is the V2.2/V2.4 proxy `|video_dur − audio_dur|`, measured with
`ffprobe -select_streams v:0/a:0 -show_entries stream=duration` (stream
durations).

| stage | file | video_dur | audio_dur | audio start | abs drift |
|-------|------|-----------|-----------|-------------|-----------|
| engine | `…/.scratch/recording-*/sources/screen.mp4` | 25.620 | 25.643 | 0.000 | **22.7 ms** (audio longer) |
| composite | `…/.scratch/recording-*/recording-*.mp4` | 25.620 | 25.556 | 0.000 | **64 ms** (audio shorter) |
| saved | `~/Movies/Zeigen/recording-*.mp4` | 25.620 | 25.560 | 0.000 | **60 ms** |

- **Engine: 22.7 ms** — inside the V2.2 acceptance range (per-display medians
  3.7 / 16.3 ms, max 73 ms). Engine sync is fine for this take.
- **Composite adds ~40 ms.** The compositor shifts audio earlier by
  `audio_shift = max(0, webcam_dur − screen_dur) + audio_start` (`composite.rs:322-323`),
  applied as `-itsoffset` (`composite.rs:345-348`); the mp4 muxer normalizes the
  negative PTS by dropping that much audio off the front. For this clip
  (format durations: screen 25.660, webcam 25.767, audio start 0.000) the shift
  was **107 ms**, dropping ~87 ms of audio (AAC-frame-quantized) — exactly the
  25.643 → 25.556 audio-duration change.
- **Save (NR + transcode) is negligible: ~4 ms** (25.556 → 25.560).
- **Audio start_time = 0.000 at every stage.** Confirms the R2 AAC-priming
  residual is clean end-to-end (see R2 note below).

### The trap for V2.4's parity gate

V2.4 verification matrix item 1 ("V−A drift across 10+ recordings … every
take's `|V − A_duration|` < 17ms", `V2-PHASE-PLAN.md`) does not name the file to
measure, and item 3 points at the saved `~/Movies/Zeigen/` output. A naive
implementation measures the **saved webcam composite** — which tests the
compositor's deliberate webcam-lag heuristic, not engine sync. That number is
governed by the noisy webcam-vs-screen *duration delta* (~107 ms here), so it
will not pass <17 ms on webcam composites **even when the engine is fine**.

Duration-drift (`|video_dur − audio_dur|`) is the wrong proxy once a webcam
composite is in the path: the compositor *intentionally* de-matches durations to
improve perceptual sync, so a larger duration-drift can mean *better* real
alignment, not worse. The proxy and the goal diverge.

### What V2.4 should do

- **Engine-sync parity:** measure the engine output —
  `…/.scratch/recording-*/sources/screen.mp4` (screen video + mic audio, no
  compositor) — against the <17 ms gate. That is the apples-to-apples comparison
  to v1.0 and to the V2.2 acceptance numbers.
- **Composite parity:** if the saved composite must be gated, use an
  offset/perceptual metric (e.g. measured A/V lag against a known visual+audio
  event), not raw duration-drift. Document the expected composite duration
  de-match as by-design.
- Either way, **state the measurement target explicitly** in the V2.4 matrix so
  the gate measures what it intends to.

### Compositor's own flagged limitation (do not re-derive)

`composite.rs:316-321` already documents that the file-duration delta can't
distinguish *webcam-started-late* from *webcam-stopped-early*; the
webcam-stopped-early case (~10% in prior testing) is under-compensated and still
feels slightly out of sync. The proper fix needs the engine to write per-pipeline
first-sample wall-clock timestamps to a sidecar the compositor can read — a
candidate V2.4+ improvement, not a V2.3 item.

### Scope guards

- **R2 unchanged.** R2 is the AAC-priming residual; this probe *confirms* it
  clean (audio start_time 0.000 at every stage). No risk-register change.
- **`composite.rs` untouched.** The compositor is working as designed and is
  charter-out-of-scope for V2.3. This note records a measurement-methodology
  finding for V2.4, not a fix.
- **No V2.3 regression.** The compositor predates V2.3; c2 (lifecycle) did not
  touch it.
