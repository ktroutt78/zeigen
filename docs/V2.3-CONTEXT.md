# V2.3: Tauri-Rust IPC integration — Context

**Gathered:** 2026-05-27
**Status:** Ready for planning
**Source of truth for scope:** `docs/V2-CHARTER.md`, `docs/V2-PHASE-PLAN.md` (V2.3 sketch), `docs/V2.2-REPORT.md`, `docs/IPC-SPEC.md`

## Phase boundary

V2.2 shipped the Track B engine binary and widened the IPC error-code surface (`CLOCK_MISMATCH`, `MIC_NO_FIRST_SAMPLE`, `MIC_SESSION_FAILED`) but stopped at the engine's external interface. V2.3 wires that interface the rest of the way into the running app and clears the V2.3 backlog the lifecycle incidents created. It is the first V2 phase to touch `src-tauri/src/` and `src/` (V2.1/V2.2 were spike/engine-only).

The defining discovery of context-gathering: **the IPC pipe already exists end-to-end.** `engine.rs` is a pure pass-through — every engine stdout line becomes a Tauri `engine-event` (`engine.rs:65-76`), no Rust code parses error codes. The frontend already renders every code via `setError(\`${ev.code}: ${ev.message}\`)` (`App.tsx:858-867`), and the `error` event types `code` as a plain `string` (`App.tsx:738`), so all three new codes already reach the user today as raw `CODE: message` text with zero new code. V2.3's error work is therefore **frontend-side legibility and behavior**, not pipe-building; the Rust IPC layer stays a pass-through.

V2.3 does NOT close the V2-CHARTER DONE bar or the parity gate (`abs_drift_ms < 17ms`, full v1.0 surface parity) — that is V2.4. V2.3's job is correct, legible IPC integration of the V2.2 codes plus the process-lifecycle backlog.

In scope:
- Frontend error-code legibility: human-readable messages in the existing error banner, raw codes preserved in logs (D-01).
- `MIC_SESSION_FAILED` partial-recording salvage, gated on clean webcam compositing (D-02).
- `MIC_DROPPED` decision resolved: stays stderr-only, no IPC event (D-03).
- Orphan webcam-ffmpeg parent-death protection (D-04, BACKLOG-V2.3-1).
- Scratch-sweeper gap-closure — per-stale-dir logging + harness self-cleanup; the sweeper itself already exists (D-05, BACKLOG-V2.3-2).
- Acceptance-run provenance: committed harness path + historical record (D-06, BACKLOG-V2.3-3).
- R6 clean-machine permission GUI test against the packaged app (D-07).
- `docs/V2-RISKS.md` updates (R3 downgrade, R6 close/update), `docs/V2.3-REPORT.md`.

Out of scope (hard line per V2-CHARTER §"Out-of-scope"):
- Engine-binary internals (`src-tauri/recording-engine/`) — V2.2 is done; V2.3 adds no engine code. D-03 explicitly declines an engine-side `MIC_DROPPED` event; D-05 puts the sweeper in Rust, not the engine.
- Enumerated UI surfaces — picker, bubble, countdown, identify overlays, review window, export panel. The error banner and the error-routing logic are not enumerated surfaces; no new components are added; the review window is reused unchanged (D-02).
- Save / edit pipeline (`src-tauri/src/edit.rs`, `save_recording`) — invariant. D-02 salvage reuses the existing `recording_finalize` → `openReview` handoff, not the save/edit path.
- New IPC events or fields — D-03 declines the only candidate. The error-code `code` values V2.2 added are the entire contract surface; V2.3 adds none.
- V2.4 parity-gate work — drift matrix, AAC priming residual close, full surface-parity walk.
- App signing / notarization / DMG installer — the deferred DMG-installer backlog item stays deferred. D-07 builds a local unsigned bundle only, the minimum for `com.zeigen.app` TCC attribution.

## Implementation decisions

### D-01: Error-code legibility — friendly messages in the existing banner (GA1)

The Rust IPC layer stays a pure pass-through; the V2.3 surface is the frontend `engine-event` handler (`App.tsx:858-867`). Map each error `code` to a human-readable string shown in the existing error banner; keep the raw `code` in the log line (and stderr, already captured by `engine.rs:78-83`). No new UI surface — the banner is not one of the charter's enumerated untouchable surfaces, and this is wiring (mapping codes the engine already emits), not a new component.

Covers all codes the user can realistically hit, not just the three new ones: the existing `PERMISSION_DENIED`, `MIC_NOT_FOUND`, `OUTPUT_PATH_INVALID`, `WRITER_FAILED`, etc., plus `CLOCK_MISMATCH`, `MIC_NO_FIRST_SAMPLE`, `MIC_SESSION_FAILED`. A code without a mapping falls back to the current `CODE: message` form so an unmapped engine code is still legible.

`code` is typed `string` (`App.tsx:738`), so the three new codes already typecheck — no type-union change is forced. The mapping is a lookup, not a switch refactor.

### D-02: `MIC_SESSION_FAILED` partial-recording salvage, gated on clean compositing (GA2)

The engine *deliberately finalizes* a partial file on `MIC_SESSION_FAILED` (V2.2-CONTEXT D-04: "finalizes whatever has been written so far"). Today the frontend `error` path discards it — `recording_cleanup_local` (`lib.rs:278`) drops the handle without calling `recording_finalize`, orphaning the partial screen capture in scratch. Salvage routes `MIC_SESSION_FAILED` (and *only* that code) to the existing `recording_finalize` → `openReview` flow instead of the discard path, so the user gets the partial recording with whatever audio survived.

**Code-scoped salvage.** Only `MIC_SESSION_FAILED` salvages. The other error codes keep the discard path because nothing useful was captured: `CLOCK_MISMATCH` fires before recording begins (0-byte output), `MIC_NO_FIRST_SAMPLE` fires ~250ms in. Salvaging those would open an empty/near-empty review.

**Hard DONE-WHEN gate (c-level, not a PLAN aside).** Salvage counts only if the finalized partial opens in review intact. The risk is the partial screen track having a short/odd length that breaks webcam compositing (`composite::composite`, `composite.rs`). The gate is exercised, not asserted: trigger `MIC_SESSION_FAILED` through the full Tauri app **with webcam active** (`ZEIGEN_FORCE_MIC_SESSION_ERROR=1` propagates to the spawned engine via inherited env — `engine.rs` `Command::new` inherits parent env), confirm the partial composites cleanly and the review window opens with a playable file.
- Pass → salvage ships for `MIC_SESSION_FAILED`.
- Fail (composite breaks on the partial) → **fall back to discard (A) for this code**, documented in the report. No half-salvage that produces a broken review.

The webcam-inactive path (no segments) is the trivial case — `recording_finalize` uses the screen mp4 directly with no composite; verify it opens too, but the load-bearing gate is the webcam-active composite.

This reuses `recording_finalize` and `openReview` unchanged — no save/edit-pipeline change, no review-window change. The new logic is purely the error-path branch in `App.tsx`.

### D-03: `MIC_DROPPED` stays stderr-only — no IPC event (GA3, resolves D-07 deferral)

V2.2 D-07 deferred to V2.3 the question "should `MIC_DROPPED` become an IPC event so the UI can surface mid-record mic-drop." **Answer: no, for V2.3.** `MIC_DROPPED` is non-fatal — the recording continues video-only (USB cessation, via D-15 cap-release) or full-length-with-silence (Continuity silence-substitution); the file is never lost. Keep the engine's stderr-only logging (`MIC_DROPPED reason=<interrupted|sample-cessation>`); add no IPC event.

Rationale, three reinforcing reasons:
1. **No forcing function.** The contract widens only when forced (Charter; R3). The recording surviving means there is no failure to report through the fatal `error` event, and the `error` event's frontend semantics ("recording ended, go idle") would be wrong for a non-fatal continuation.
2. **A new event is structural widening.** Unlike V2.2's new `code` string values (no shape change), a `mic_dropped`/`warning` event is a new message type — it triggers the ADR per V2.2-CONTEXT §"ADR slot" and is the heavier class of change R3 guards.
3. **No surface to host it.** Surfacing a mid-record warning needs a non-blocking indicator (toast/badge) that does not exist; building one is a new UI component = charter violation. The honest answer to D-07's "does the UI need to surface it" is "the UI *can't* without a UI phase to host the indicator."

Recorded as a deferred idea (event shape + UI affordance) for the future UI phase. No engine change, no Rust change, no frontend change — D-03 is a decision to *not* widen, documented.

### D-04: Orphan webcam-ffmpeg protection — parent-PID poll, kqueue fallback (GA4, BACKLOG-V2.3-1)

The 187GB incident: a webcam-capture ffmpeg kept writing for 5 days after its parent died. The webcam ffmpeg is spawned by the **Tauri/Rust process** (`webcam.rs:37-62`), not the Swift engine — the backlog's "child of the engine" wording is imprecise; the death-signal must track the **Tauri process PID**. The existing `Drop` impl (`webcam.rs:90-97`) kills the child on normal teardown but does not run on SIGKILL / crash / force-quit, which is exactly the orphan scenario.

macOS has no `PR_SET_PDEATHSIG` (Linux-only), so there is no one-liner. The death-signal **must live in a separate process** from Rust — a Rust-side supervisor thread dies with the Rust process under SIGKILL and cannot clean up.

**Mechanism: parent-PID poll (primary).** Spawn the webcam ffmpeg under a minimal wrapper that self-terminates ffmpeg when its parent (the Tauri process) dies — detected by reparenting (`getppid()` becoming 1 / no longer the original Tauri PID), polled at ~1s. When Tauri is SIGKILLed, the wrapper reparents to launchd, the next poll observes it, and the wrapper kills ffmpeg. This is the simplest mechanism that actually stops an actively-writing orphan, per CLAUDE.md simplicity.

**Fallback: kqueue `EVFILT_PROC` parent-watch (if polling proves flaky).** A wrapper that registers a kqueue watch on the original parent PID and kills ffmpeg on the parent's `NOTE_EXIT`. Event-driven, lower latency, more code. Adopt only if the poll exhibits a real problem (latency, missed reparent) during c2.

The existing `Drop`-kill stays as the normal-exit fast path; the wrapper covers crash/SIGKILL. **The sweeper (D-05) does not substitute for this** — the sweeper is an after-the-fact backstop that runs on the *next* app launch; it cannot stop an orphan that is actively writing while the app stays down (the 5-day write happened because the app was not restarted). Only the death-signal stops the bleed.

### D-05: Scratch-sweeper already exists — scope is gap-closure (GA5, BACKLOG-V2.3-2)

**Corrected 2026-05-27 (follow-up to the CONTEXT checkpoint): the sweeper this item asked for already exists and is already wired, at the exact layer GA5 picked.** `sweep_stale_scratch()` (`lib.rs:425-434`) calls `thumbs::sweep_dir_older_than(root, 24h)` (`thumbs.rs:121-139`) inside the Tauri `.setup()` closure (`lib.rs:565`) — it runs on every app launch, Rust-side, 24h threshold, deleting anything in `~/Movies/Zeigen/.scratch/` older than 24h. BACKLOG-V2.3-2's "add it in main.swift" was filed without checking that Rust already does this; the GA5 pick (Rust app startup, not the engine) is the *implemented reality*, not a change to make. The production daily-driver leak is therefore **already mitigated on main**.

D-05's real scope is closing the two gaps the existing sweeper leaves (option B from discussion — small enhancement, no rebuild):

- **(a) Per-stale-dir warning logging.** The existing sweep is silent ("Best-effort; failures are silent" — `lib.rs:421-424`). The backlog's literal spec wants a warning per stale dir so a cleaned zombie leaves a breadcrumb. Add a log line per removed dir (path + age) in `sweep_stale_scratch` / `sweep_dir_older_than`.
- **(b) Harness self-cleanup for the dev-invocation path.** Harness / UAT / direct-engine runs never launch the Tauri app, so the existing sweeper never runs for them — exactly the accumulation that started this incident in dev. The harness owns its own scratch: clean up on teardown. Per GA5, the engine does not reach into the app's Movies dir.

Not addressed (deliberately): the existing sweeper deletes *all* entries >24h rather than filtering `recording-*`. `.scratch` only ever holds `recording-*` dirs, so the broader sweep is equivalent and needs no narrowing.

**D-04/D-05 distinction stays sharp — the sweeper is residue cleanup only, NOT an orphan stopper.** A directory's mtime does not update when a file *inside* it is written, so the sweeper can `remove_dir_all` a `recording-*` dir whose `webcam-00.mp4` is still being written by an orphan ffmpeg — which only unlinks an inode the process still holds open. The orphan keeps writing to the now-unlinked file and the disk space is not reclaimed until the process dies. **D-04 (kill the orphan) is the load-bearing fix for the 187GB class; D-05's sweeper cannot stop an actively-writing orphan and never could.**

### D-06: Acceptance-run provenance — committed path, no backfill (GA6, BACKLOG-V2.3-3)

The c2 acceptance drift numbers survive only in commit `ac5e115`'s body — `harness/v2-drift/output/` is gitignored (`.gitignore:37-38`) and the endSession-removal control run overwrote the raw acceptance CSV. Authoritative data a later phase depends on (V2.4's R2 gate-close reads the c2 residual) should be reproducible from the repo.

**Fix: a committed acceptance path, separate from throwaway runs.**
- Create a tracked `harness/v2-drift/acceptance/` directory. The harness writes acceptance runs there (a flag or distinct output target); ad-hoc/experimental runs continue to land in the gitignored `output/`.
- `.gitignore` keeps `output/*.mp4` / `output/*.csv` ignored and ensures `acceptance/` is tracked.
- V2.4's fresh acceptance runs land in `acceptance/` natively.

**No backfill / no re-run.** The lost c2 run cannot be reproduced bit-for-bit (machine state differs run to run), and V2.4's R2 decision needs the residual *magnitude*, which the `ac5e115` commit-body table already preserves. Copy that table verbatim into a committed file in `acceptance/` (CSV or markdown) as the historical record, annotated with its provenance (reconstructed from commit `ac5e115`, raw run lost). Re-running would produce different numbers, not the original.

### D-07: R6 clean-machine permission test against the packaged app (GA7)

R6 (permission flow) is partially realized: the engine's pre-request order is code-verified (`main.swift` does `CGRequestScreenCaptureAccess()` → `AVCaptureDevice.requestAccess(for: .audio)` → `.ready`), but the live clean-machine GUI run was deferred to V2.3 because TCC attributes screen/mic permission to the *responsible process*, and against the CLI dev build that is the host terminal/IDE, not a Zeigen bundle. The meaningful test needs `com.zeigen.app` as the responsible process.

**Packaging scope: local unsigned bundle only.** Produce a `tauri build` `.app` with bundle id `com.zeigen.app` — the minimum for correct TCC attribution. No signing, no notarization, no DMG (the DMG-installer backlog item stays deferred). The unsigned local bundle is sufficient to characterize the permission sequence and the relaunch behavior under the real bundle id.

**Execution: scripted prep + manual GUI run.** This is a human-GUI UAT (System Settings dialogs, permission grants) — not harness-automatable. A small prep script resets TCC for `com.zeigen.app` (Screen Recording + Microphone) and launches the bundle; the user observes and grants. Deliverable is a written UAT procedure plus the run.

**Safety: explicit confirm before any `tccutil reset`.** A `tccutil reset` mutates the dev machine's permission state and is mildly destructive — confirm with the user before running it. The daily driver (`v1.0` tag) uses different attribution, but confirm scope before resetting.

**Re-check the relaunch gotcha.** V2.2 recorded that a Screen Recording grant may not take effect until the responsible process relaunches, flagged as a possible dev-build artifact. D-07 characterizes it against the real `com.zeigen.app` bundle: does the grant take effect live, or require a relaunch? Record the finding in the R6 update.

### Established patterns the integration preserves

- **Engine self-resets to idle on any `error`** (`App.tsx:858-867` comment). The salvage branch (D-02) must respect this — on `MIC_SESSION_FAILED` the engine has already returned to idle, so the frontend routes to finalize without sending `Stop` (which would produce a follow-on `INVALID_STATE`, exactly the trap `recording_cleanup_local` was built to avoid).
- **Tauri IPC event/return ordering caveat** (`lib.rs:365-369`) — the existing finalize path already guards against out-of-order composite-progress; the salvage path reuses it as-is.
- **Engine inherits parent env** — `engine.rs` `Command::new` does not scrub env, so `ZEIGEN_FORCE_MIC_SESSION_ERROR=1` set before launching the app propagates to the spawned engine (D-02 test mechanism).

### Integration points

- The error path is the only state-machine seam V2.3 touches: `error` event → (salvage | discard) → idle. Everything else (`started`/`progress`/`stopped`/`paused`/`resumed`) is unchanged.
- The webcam ffmpeg lifecycle (`webcam.rs` + `recording_finalize`/`recording_cleanup_local`) is the D-04 seam — the wrapper changes how the child is spawned, not how it is stopped on the normal path.
- The packaged-app build (D-07) is the first time V2 produces a `com.zeigen.app` bundle; it exercises the real Tauri spawn path for the engine (distinct from the harness's direct-engine spawn), which is also where the R3 end-to-end claim is finally validated.

## ADR slot
